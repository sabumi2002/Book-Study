# 팩토리패턴
- 용도
	- 구상클레스를 많이 사용하면 새로운 구상클래스가 추가될 때마다 코드를 고쳐야 하므로 수많은 문제가 생길 수 있다. (변경에 닫힌 코드)
	- 새로운 구상 형식을 써서 확장해야 할 때는 인터페이스에 맞춰서 코딩하면 시스템에서 일어날 수 있는 여러 변화에 대응할 수 있다. (팩토리 = 다형성)
	
- 팩토리패턴 = 캡슐화
	- 모든 팩토리 패턴은 객체 생성을 캡슐화한다.
	- 팩토리 메소드 패턴은 서브 클래스에서 어떤 클래스를 만들지 결정함으로써 객체 생성을 캡슐화한다.
	- 예를 들어 PizzaStore에서 어떤 피자를 만들지 선택한다면 그 피자를 만드는 모든 방법을 캡슐화한다.

![[Pasted image 20251210163700.png]]

- 팩토리메소드 패턴 정의
	- 객체를 생성할 때 필요한 인터페이스를 만든다.
	- 어떤 클래스의 인스터스를 만들지 서브클래스에서 결정한다.
	- 팩토리 메소드 패턴을 사용하면 클래스 인스턴스 만드는 일을 서브클래스에게 맡기게 된다.


- 의존성 뒤집기
	- PizzaStore를 구현해야한다.
	- 하지만 PizzaStore가 구상 피자 형식을 알면 구상 클래스에 의존해야하는 문제가 발생한다.
	- 추상화된 Pizza클래스를 만들고 동일한 인터페이스를 공유해야한다.
	- PizzaStore에서 구상클래스를 없애려면 팩토리를 사용해야한다.
	- 팩토리를 사용하면 다양한 구상 피자 형식이 추상화된 Pizza 클래스에 의존할 수 있다.
	- 마찬가지로 피자가게도 추상화된 Pizza 클래스에 의존하게 된다.
	- 처음엔 PizzaStore가 구상 클래스인 Pizza 들을 의존하는 구조였지만 PizzaStore, Pizza 모두 추상화된 Pizza 클래스 하나만 의존하게 되므로 의존성이 뒤집히게 된다.


- 추상팩토리 패턴 정의
	- 구상클래스에 의존하지 않고도 서로 연관되거나 의존적인 객체로 이루어진 제품군을 생산하는 인터페이스를 제공한다.
	- 구상클래스는 서브클래스에서 만든다.

# 싱글턴 패턴
- 용도
	- 하나만 있어도 충분히 돌아가는 객체(스레드 풀, 캐시, 대화상자, 사용자 설정, 레지스트리 설정, 로그 기록용 객체 등)를 쓸 때 인스턴스가 2개 이상이면 프로그램이 이상하게 돌아간다던가, 자원을 불필요하게 잡아먹는다던가, 결과에 일관성이 없어진다던가 하는 심각한 문제가 발생할 때 쓰임

- 싱글턴 패턴 정의
	- 싱글턴 패턴은 클래스 인스턴스를 하나만 만들고, 그 인스턴스로의 전역 접근을 제공한다.
	- 주의점
		- 싱글턴 패턴을 실제로 적용할 때는 클래스에서 하나뿐인 인스턴스를 관리하도록 만들면 된다. 그리고 다른 어떤 클래스에서도 자신의 인스턴스를 추가로 만들지 못하게 해야 한다. 인스턴스가 필요하다면 반드시 클래스 자신을 거치도록 한다.
		- 어디서든 그 인스턴스에 접근할 수 있도록 전역 접근 지점을 제공한다. 언제든 이 인스턴스가 필요하면 클래스에 요청할 수 있게 만들어 놓고, 요청이 들어오면 그 하나뿐인 인스턴스를 건네주도록 만들어야한다. 싱글턴이 게으른 방식으로 생성되도록 구현할 수 있다. 특히 자원을 많이 잡아먹는 인스턴스가 있다면 이런 기법이 꽤 유용한다.
