# 유사한 클래스 통합하기
메서드가 상수를 반환할 때 그것을 상수 메서드라고 부릅니다. 경우에 따라 다른 값을 반환하는 상수 메서드를 공유하기 때문에 이 두 클래스는 합칠 수 있습니다. 이와 같이 두 개의 클래스를 합치는 것은 두 단계로 이루어지는데, 절차는 분수를 더하는 알고리즘과 유사합니다. 분수를 더하는 첫 번째 단계는 분모를 동일하게 만드는 것이고 클래스를 견합하는 첫 번째 단계는 상수 메서드를 제외한 클래스의 모든 것을 동일하게 만드는 것입니다. 분수를 더하는 두 번째 단계는 실제 더하기입니다. 클래스의 경우에는 실제로 합치는 것입니다.

### 리팩터링 패턴: 유사 클래스 통합
#### 설명
일련의 상수 메서드를 공통으로 가진 두 개 이상의 클래스에서 이 일련의 상수 메서드가 클래스에 따라 다른 값을 반환할 때마다 이 리팩터링 패턴을 사용해서 클래스를 통합할 수 있습니다. 여기서 이렬ㄴ의 상수 메서드 집합을 기준(basis)이라고 합니다. 일련의 상수 메서드가 두 개일 때 **두 개의 접점을 가진 기준**이라고 합니다. 우리는 가능한 한 적은 메서드를 가진 기준을 원합니다. X개의 클래스를 통합하려면 최대 (X-1)개의 접점을 가진 기준이 필요합니다. 클래스의 수가 적어진다는 것은 일반적으로 더 많은 구조를 발견했다는 것을 의미하므로 클래스를 통합하는 것은 좋은 것입니다.

### 조건을 위한 산술 규칙 사용
조건표현식이 무엇을 하는지 알지 못하더라도 조작할 수 있습니다. ||(및 |) dms +(더하기) 처럼 동작하고 &&(및 &) 은 x(곱하기) 처럼 동작합니다. 이것을 기억하는 방법은 ||으로 +를 만들 수 있으며 &의 내부에 x 가 숨겨져 있다고 연상하는 것입니다. 이것은 || 주위를 괄호로 묶는데 도움이 되며 모든 정규 산술 법칙이 적용됩니다.
이 규칙을 사용할 수 있으려면 항상 **순수 조건 사용** 으로 부수적인 동작을 하지 않아야 합니다.

### 규칙: 순수 조건 사용
#### 정의
조건은 항상 순수 조건이어야 합니다.

#### 설명
조건은 if 또는 while 뒤에 오는 것과 for 루프의 가운데에 있는 것입니다. 순수 라는 말은 조건에 부수적인 동작이 없음을 의미합니다. 부수적인 동작이란 조건이 변수에 값을 할당하거나 예외를 발생시키거나 출력, 파일 쓰기 등과 같이 I/O 와 상호작용하는 것을 의미합니다.
순수한 조건을 갖는 것은 여러 가지 이유로 중요합니다. 첫째, 부수적인 동작이 존재하는 조건으로 인해 앞서 언급한 규칙들을 사용할 수 없습니다. 둘째, 부수적인 동작은 조건문에서 흔하게 사용하지 않기 때문에 조건에 부수적인 동작이 있을 것으로 예상하지 않습니다. 이 말은 곧 그것들을 찾아내야 한다는 것을 의미하며, 어떤 조건이 부수적인 동작을 가지고 있는지를 추적하는데 더 많은 시간과 노력을 투자해야 한다는 것을 암시합니다.

다음과 같은 코드는 일반적이며 readLine 은 다음 줄을 반환하고 포인터를 전진시킵니다. 포인터를 전진시키는 것은 부수적인 동작이므로 이 조건은 순수하지 않습니다. 오른쪽에 있는 더 나은 구현은 줄을 가져오고 포인터를 이동하는 책임을 분리합니다. null을 반환하는 대신 더 읽을 것이 있는지 여부를 확인하는 메서드를 도입하는 것이 더 좋습니다.

readLine은 부수적인 동작 없이 원하는 만큼 호출할 수 있습니다.

우리가 구현을 통제할 수 없으므로 부수적인 동작에서 반환을 분리할 수 없을 경우 캐시를 사용할 수 있습니다. 캐시를 구현하는 방법에는 여러 가지가 있기 때문에 구현에 대해 자세한 설명 없이 모든 메서드에서 사용할 수 있고 부수적인 동작을 반환 부분과 분리할 수 있는 범용 캐시를 소개하겠습니다.
#### 스멜
이 규칙은 '명령에서 질의 분리(Separate queries from commands)'라는 일반적인 스멜에서 비롯합니다. 이 스멜은 알아채기가 어렵지 않습니다. 스멜에서 '명령'은 부작용이 있는 모든 것을 의미하고 '질의'는 순수한 것을 의미합니다. 이 스멜을 따르는 쉬운 방법은 void 메서드에서만 부수적인 동작을 허용하는 것입니다. 부수적인 동작을 하거나 무언가를 반환하는 것 중 하나만 하게 됩니다.

일반적인 스멜과 이 규칙의 유일한 차이점은 정의보다는 호출에 초점을 맞춘다는 것입니다. Design by Contract: by Example 책에서 미첼과 맥킴은 모든 경우에 엄격한 분리를 따르는 더 많은 원칙을 세웠습니다. 이 책에서는 조건에 초점을 맞추기 위해 이 스멜을 순화했습니다. 왜냐하면 조건 이외에서 질의와 명령을 혼합하는 것은 우리의 리팩터링 결과에 영향을 주지 않기 때문입니다. 스멜에 집착하는 것은 아마도 스타일의 문제일 것입니다. 또한 메서드는 무엇인가를 반환하고 변경하는 것이 더 일반적이기 때문에 이를 찾는 것은 어렵지 않습니다. 

#### 의도
목적은 데이터를 가져오는 것과 변경하는 것을 분리하는 것입니다. 이는 코드를 더 깔끔하고 예측가능하게 만듭니다. 메서드가 더 간단하기 때문에 일반적으로 더 나은 이름을 지정할 수 있습니다. 부수적인 동작은 위험한 전역 상태의 일종으로 볼 수 있는데, 이것은 2장에서 언급했듯이 위험합니다. 따라서 이것을 격리하면 관리하기가 더 쉬워집니다.

# 클래스 간의 코드 통합
다음은 전략패턴의 도입입니다.

### 클래스 관계를 묘사하기 위한 UML 클래스 다이어그램 소개
종종 아키텍처나 일이 일어나는 순서 같은, 코드에 대한 속성을 전달할 필요가 있습니다. 이러한 속성 중 일부는 다이어그램으로 더 쉽게 전달할 수 있습니다. 이럴때 사용하는 것이 UML 클래스 다이어그램입니다.

UML은 코드에 대한 속성을 전달하기 위해 다양한 유형의 표준 다이어그램으로 구성됩니다. 몇 가지 예로는 시퀀스 다이어그램, 클래스 다이어그램, 활동 다이어그램이 있습니다.

그 중 클래스 다이어그램은 인터페이스와 클래스의 구조가 서로 어떤 관계가 있는지 보여줍니다. 우리는 상자, 제목, 때때로 메서드로 클래스를 표현하지만 필드는 거의 없습니다. 인터페이스는 클래스처럼 표시되지만, 제목 위에 interface라고 표시합니다. 메서드와 필드가 private 인지 public 인지를 나타내기도 합니다. 

### 리팩터링 패턴: 전략 패턴의 도입
#### 설명
다른 클래스를 인스턴스화해서 변형(variance)을 도입하는 개념을 전략 패턴이라고 합니다. 일반적으로 앞에서 보여준 것과 유사한 클래스 다이어그램으로 설명됩니다.

많은 패턴이 전략 패턴의 다른 형태입니다. 전략이 필드를 가지고 있는 경우 이를 상태 패턴(state pattern) 이라고 합니다. 정확한 이름을 아는 것이 의사소통에 많은 도움이 되진 않습니다. 여기선 큰 범주로 전략패턴이라는 용어를 사용해서 코드를 자체 클래스로 옮기는 것을 설명합니다. 이러면 새로운 변형을 추가하지 않더라도 확장성을 가질 수 있습니다.

이것은 타입을 가리키는 코드를 클래스로 변환하는 것과 다릅니다. 이러한 클래스는 데이터를 나타내므로 많은 메서드를 해당 클래스에 밀어 넣는 경향이 있습니다. 전략 클래스가 완료된 후 메서드를 추가하는 경우는 거의 없습니다. 대신 기능을 변경해야 하는 경우 새로운 클래스를 만드는 것을 선호합니다. 

변형(전략)은 전략 패턴의 목적이기 때문에 항상 상송으로 묘사됩니다. 일반적으로 인터페이스에서 상속받지만 때때로는 추상클래스에서도 상속을 받습니다. 그러나 이미 그 단점에 대해 논의했기 때문에 추상 클래스에서의 상속은 사용하지 않습니다.

두 가지 상황에서 전략 패턴을 도입합니다. 첫째, 코드에 변형을 도입하고 싶어서 리팩터링을 수행하는 경우입니다. 이 경우 결국 인터페이스가 있어야 합니다. 그러나 이 리팩터링을 최대한 빠르게 하려면 인터페이스는 나중에 만드는 것이 좋습니다. 둘째, 떨어지는 성질을 코드화했던 상황에서 바로 변형의 추가가 필요하다고 예상하지 않았을 때입니다. 단지 클래스 간의 동작을 통합하려는 경우입니다. 구현체가 하나뿐인 인터페이스를 만들지 말 것 이라는 규칙이 있습니다. 인터페이스가 필요할 때 바로 혹은 나중에 구현에서 인터페이스 추출 이라는 리팩터링 패턴을 사용합니다. 


#### 절차
1. 분리하려는 코드에 대해 메서드 추출을 수행합니다. 다른 것과 통합하려면 메서드가 동일한지 확인합니다.
2. 새로운 클래스를 만듭니다.
3. 생성자에서 새로운 클래스를 인스턴스화합니다.
4. 메서드를 새로운 클래스로 옮깁니다.
5. 필드에 종속성이 있는 경우 다음을 수행합니다.
	1. 필드를 새로운 클래스로 옮기고 옮긴 필드에 대한 접근자를 만듭니다.
	2. 새로운 접근자를 사용해서 원래 클래스에서 발생하는 오류를 바로잡습니다.
6. 새로운 클래스의 나머지 오류에 대해 해당 값을 대체할 매개변수를 추가합니다.
7. 메서드의 인라인화를 사용해서 1단계의 추출을 반대로 합니다.
