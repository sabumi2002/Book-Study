# 나쁜 코드를 식별 가능하게 만들기

좋은 코드를 만드는 일만큼 중요한 것이 있습니다. 바로 **나쁜 코드를 숨기지 않는 것**입니다.

13장의 핵심은 조금 도발적입니다. 시간이 부족해서 코드를 제대로 리팩터링할 수 없다면, 어설프게 깨끗한 척 만들지 말고 문제를 식별 가능하게 남기라는 것입니다. 책에서는 이를 **안티-리팩터링(anti-refactoring)** 이라고 부릅니다.

이 말은 일부러 코드를 망치자는 뜻이 아닙니다. 문제 있는 코드를 그럴듯하게 포장해서 팀이 알아차리지 못하게 만드는 것이 더 위험하다는 뜻입니다.

## 왜 나쁜 코드를 숨기면 안 될까?

실무에서는 항상 충분한 시간이 주어지지 않습니다. 기능은 급하고, 요구사항은 바뀌고, 테스트는 부족하고, 기존 코드는 이미 복잡합니다. 이때 우리는 종종 "조금만 정리해서 나쁘지 않게" 만들고 넘어가고 싶어집니다.

문제는 여기서 시작됩니다.

나쁘지 않아 보이는 코드는 다시 발견되기 어렵습니다. 겉으로는 괜찮아 보이기 때문에 다음 개발자는 그 코드가 리팩터링 대상이라는 사실을 놓치기 쉽습니다. 반대로 명확히 나쁜 코드는 신호가 됩니다.

> 잘 만들 수 없다면, 적어도 문제를 보이게 만들어야 한다.

나쁜 코드를 눈에 띄게 두면 두 가지 효과가 있습니다.

- 나중에 다시 찾기 쉽다.
- 현재 개발 속도나 품질 관리 방식이 지속 가능하지 않다는 신호를 준다.

이 지점에서 중요한 것은 심리적 안정감입니다. 팀이 문제를 드러낸 사람을 비난한다면 개발자는 코드를 숨기기 시작합니다. 반대로 문제를 드러내도 안전한 팀이라면, 나쁜 코드는 개선의 출발점이 됩니다.

## 깨끗한 코드와 레거시 코드를 분리하기

책은 코드베이스를 애매한 중간 상태로 두는 것을 경계합니다.

- 아주 좋은 코드
- 나쁘지 않은 코드
- 명확히 나쁜 코드

이 셋 중 가장 위험한 것은 때로 "나쁘지 않은 코드"입니다. 명확히 나쁜 코드는 눈에 띄지만, 나쁘지 않은 코드는 문제를 숨깁니다.

예를 들어 다음 코드는 언뜻 보기에는 괜찮아 보입니다.

```java
void applyDiscount(Order order, boolean vip) {
    if (vip) {
        order.discount(20);
    } else {
        order.discount(5);
    }
}
```

짧고 읽기 어렵지도 않습니다. 하지만 `boolean vip`는 도메인 개념을 숨깁니다. 할인 정책이 `VIP`, `WELCOME`, `SEASONAL`처럼 늘어나면 이 구조는 곧 무너집니다.

차라리 아래처럼 문제를 드러내는 편이 나을 수 있습니다.

```java
enum DiscountType {
    VIP,
    BASIC
}

void applyDiscount(Order order, DiscountType discountType) {
    if (discountType == DiscountType.VIP) {
        order.discount(20);
    } else if (discountType == DiscountType.BASIC) {
        order.discount(5);
    }
}
```

아직 좋은 코드는 아닙니다. `if` 분기가 있고, 할인 정책도 각 타입의 클래스로 분리되지 않았습니다. 하지만 `DiscountType`이 생기면서 "여기에는 타입 코드가 있고, 언젠가 다형성으로 바꿀 수 있다"는 신호가 분명해졌습니다.

이것이 13장의 관점입니다. 완벽하게 고칠 수 없다면, 다음 사람이 고칠 수 있도록 문제를 더 잘 보이게 만들어야 합니다.

## 나쁜 코드를 식별하는 세 가지 방법

나쁜 코드를 완벽하게 판별하는 공식은 없습니다. 가독성은 사람의 경험과 맥락에 영향을 받기 때문입니다. 대신 책은 나쁜 코드를 판단하는 대표적인 세 가지 기준을 소개합니다.

### 1. 이 책의 규칙

`Five Lines of Code`는 추상적인 코드 스멜 대신 구체적인 규칙을 제시합니다.

- 메서드는 다섯 줄 이내로 유지한다.
- `if`는 함수의 시작에만 둔다.
- `else`를 사용하지 않는다.
- 타입 코드를 클래스로 바꾼다.
- 공통 접사를 사용하지 않는다.

이 규칙들은 완벽한 진리가 아니라 훈련 도구입니다. 장점은 명확합니다. 경험이 부족해도 위반 여부를 비교적 쉽게 찾을 수 있습니다.

예를 들어 아래 코드는 규칙 위반이 눈에 잘 보입니다.

```java
void completePayment(Order order) {
    if (order.isPaid()) {
        return;
    }

    if (order.getAmount() > 100_000) {
        order.applyLargeOrderDiscount();
    } else {
        order.applyBasicDiscount();
    }

    order.pay();
}
```

`if`가 여러 번 등장하고, `else`도 있습니다. 이 코드는 "나쁘다"라고 단정하기보다, 리팩터링을 검토해야 한다는 신호로 볼 수 있습니다.

### 2. 코드 스멜

코드 스멜(code smell)은 코드에 문제가 있을 가능성을 알려주는 증상입니다. 대표적인 예는 다음과 같습니다.

- 중복 코드
- 긴 메서드
- 긴 매개변수 목록
- 매직 넘버
- 데이터 클래스
- 타입 코드

예를 들어 다음 코드는 매직 넘버와 중복 조건이 함께 나타납니다.

```java
boolean canUseCoupon(User user, Order order) {
    return user.getGrade().equals("VIP")
            && order.getTotalPrice() >= 30000
            && order.getItemCount() >= 2;
}
```

`"VIP"`, `30000`, `2`는 각각 도메인 규칙일 가능성이 높습니다. 이 값들이 여러 곳에 퍼지면 정책 변경이 어려워집니다.

### 3. 복잡도 지표

컴퓨터가 계산할 수 있는 지표도 있습니다.

**순환 복잡도(cyclomatic complexity)** 는 코드가 지나갈 수 있는 경로의 수를 봅니다. `if`, `for`, `while`, `&&`, `||` 같은 분기가 많을수록 값이 커집니다. 경로가 많다는 것은 테스트해야 할 경우의 수도 많다는 뜻입니다.

```java
int shippingFee(Order order) {
    if (order.isFreeShippingTarget()) {
        return 0;
    }

    if (order.isJeju()) {
        return 5000;
    }

    return 3000;
}
```

이 코드는 가능한 흐름이 여러 갈래입니다. 아직 단순하지만 조건이 계속 늘어나면 테스트와 이해 비용이 함께 증가합니다.

**인지 복잡도(cognitive complexity)** 는 사람이 코드를 읽을 때 얼마나 많은 맥락을 머릿속에 유지해야 하는지를 봅니다. 특히 중첩이 깊을수록 읽기 어려워집니다.

```java
void approve(Order order) {
    if (order.isPaid()) {
        if (order.hasStock()) {
            if (order.isAddressValid()) {
                order.approve();
            }
        }
    }
}
```

이 코드는 분기 수보다 중첩 구조가 문제입니다. 읽는 사람은 `paid`, `stock`, `address` 조건을 계속 기억해야 합니다.

## 나쁜 코드를 안전하게 드러내는 세 가지 원칙

나쁜 코드를 눈에 띄게 만든다고 해서 아무렇게나 망가뜨려도 되는 것은 아닙니다. 책은 세 가지 원칙을 제시합니다.

### 1. 올바른 정보를 훼손하지 않는다

좋은 이름, 정확한 주석, 의미 있는 타입 정보는 보존해야 합니다. 문제가 있는 코드를 더 눈에 띄게 만들겠다고 멀쩡한 정보를 없애면 안 됩니다.

나쁜 예:

```java
void process(Order order) {
    // ...
}
```

기존 메서드 이름이 `approveOrder`처럼 의미 있었는데 `process`로 바꿨다면 정보가 사라진 것입니다. 문제를 드러내는 것이 아니라 더 헷갈리게 만든 것입니다.

### 2. 향후 리팩터링을 어렵게 만들지 않는다

나쁜 코드를 표시하는 목적은 나중에 고치기 위함입니다. 따라서 다음 리팩터링을 더 쉽게 만드는 방향이어야 합니다.

예를 들어 `Map<String, Object>`로 뭉개는 것은 문제를 숨깁니다.

```java
void createOrder(Map<String, Object> values) {
    // customerId, productId, quantity, address...
}
```

차라리 긴 매개변수 목록이 더 나을 수 있습니다.

```java
void createOrder(
        Long customerId,
        Long productId,
        int quantity,
        String zipCode,
        String address
) {
    // ...
}
```

좋은 최종 형태는 아닙니다. 하지만 어떤 데이터가 필요한지 명확히 보입니다. 이후 `ShippingAddress`, `OrderLine`, `CustomerId` 같은 객체로 묶기도 쉽습니다.

### 3. 한눈에 문제를 알아볼 수 있어야 한다

표시는 신호여야 합니다. 너무 미묘하면 효과가 없습니다. 개발자가 코드를 보자마자 "여기는 리팩터링이 필요하겠는데?"라고 느낄 수 있어야 합니다.

대표적인 신호는 다음과 같습니다.

- 긴 메서드
- 긴 매개변수 목록
- 반복되는 접두사
- 매직 넘버
- 타입 코드
- 과하게 드러난 getter/setter
- 의미 있는 빈 줄과 주석

## 나쁜 코드를 눈에 띄게 만드는 방법

책에서 소개하는 방법들은 모두 "최종 설계"가 아닙니다. 리팩터링을 위한 표식에 가깝습니다.

### 타입 코드를 드러내기

`boolean`은 두 가지 상태만 표현할 때는 간단하지만, 도메인 의미를 숨기기 쉽습니다.

```java
void sendMessage(User user, boolean marketing) {
    if (marketing) {
        mailer.sendMarketing(user);
    } else {
        mailer.sendNotice(user);
    }
}
```

아래처럼 enum을 사용하면 아직 분기는 남아 있어도 의미가 분명해집니다.

```java
enum MessageType {
    MARKETING,
    NOTICE
}

void sendMessage(User user, MessageType type) {
    if (type == MessageType.MARKETING) {
        mailer.sendMarketing(user);
    } else if (type == MessageType.NOTICE) {
        mailer.sendNotice(user);
    }
}
```

다음 리팩터링 방향도 자연스럽습니다. `MessageType`별 클래스를 만들고, 발송 로직을 각 클래스로 옮기면 됩니다.

### 매직 넘버를 남길 때는 조심하기

매직 넘버는 대부분 나쁜 신호입니다. 그래서 오히려 눈에 잘 띕니다.

```java
boolean isFreeShipping(Order order) {
    return order.getTotalPrice() >= 50000;
}
```

`50000`이 무료 배송 기준이라는 사실을 모두가 안다면 괜찮아 보일 수도 있습니다. 하지만 이 값이 정책이라면 이름이 필요합니다.

```java
private static final int FREE_SHIPPING_MIN_PRICE = 50000;

boolean isFreeShipping(Order order) {
    return order.getTotalPrice() >= FREE_SHIPPING_MIN_PRICE;
}
```

다만 13장의 관점에서는 중요한 예외가 있습니다. 상수 이름이 부정확하거나 오히려 정보를 왜곡한다면, 잘못된 이름으로 숨기는 것보다 값을 드러내고 주석으로 맥락을 남기는 편이 나을 수 있습니다.

```java
boolean isFreeShipping(Order order) {
    // 2026년 상반기 프로모션 기준. 정책 확정 후 ShippingPolicy로 이동한다.
    return order.getTotalPrice() >= 50000;
}
```

핵심은 정보를 잃지 않는 것입니다.

### 주석과 빈 줄로 구조를 드러내기

일반적으로 주석은 줄이는 편이 좋습니다. 하지만 레거시 코드를 정리하는 과정에서는 주석이 리팩터링 위치를 표시하는 도구가 될 수 있습니다.

```java
void register(UserRequest request) {
    validateEmail(request.email());
    validatePassword(request.password());

    User user = new User(request.email(), request.password());
    userRepository.save(user);

    mailer.sendWelcomeMail(user);
}
```

빈 줄만으로도 세 덩어리가 보입니다.

- 검증
- 사용자 생성과 저장
- 환영 메일 발송

이 상태는 좋은 최종 코드가 아닙니다. 하지만 다음 단계가 보입니다. 각각 `validateRequest`, `createUser`, `sendWelcomeMail`로 추출할 수 있습니다.

주석도 비슷하게 사용할 수 있습니다.

```java
void register(UserRequest request) {
    // 요청 검증
    validateEmail(request.email());
    validatePassword(request.password());

    // 사용자 생성
    User user = new User(request.email(), request.password());
    userRepository.save(user);

    // 가입 완료 알림
    mailer.sendWelcomeMail(user);
}
```

이 주석들은 나중에 메서드 이름의 후보가 됩니다.

### 이름으로 묶어야 할 데이터를 드러내기

반복되는 접두사는 보통 좋지 않은 신호입니다. 하지만 캡슐화할 후보를 드러내는 데는 유용합니다.

```java
class OrderForm {
    String shippingZipCode;
    String shippingAddress1;
    String shippingAddress2;
    String billingZipCode;
    String billingAddress1;
    String billingAddress2;
}
```

`shipping`과 `billing` 접두사가 반복됩니다. 이는 `ShippingAddress`, `BillingAddress`로 분리할 수 있다는 신호입니다.

좋은 최종 형태는 다음에 가깝습니다.

```java
class OrderForm {
    Address shippingAddress;
    Address billingAddress;
}
```

반복 접두사는 제거 대상이지만, 동시에 리팩터링 방향을 알려주는 표식이기도 합니다.

### 긴 메서드와 긴 매개변수 목록을 숨기지 않기

긴 메서드는 대부분의 개발자가 바로 알아차립니다. 그래서 문제가 분명한데도 어설프게 작은 메서드 여러 개로 나누면 오히려 구조가 숨겨질 수 있습니다.

예를 들어 이름만 그럴듯한 메서드들이 실제로는 같은 상태를 계속 건드린다면, 긴 메서드 하나로 되돌려 전체 흐름을 다시 보는 편이 나을 수 있습니다.

긴 매개변수 목록도 마찬가지입니다.

```java
void reserve(
        Long userId,
        Long roomId,
        LocalDate checkIn,
        LocalDate checkOut,
        int adultCount,
        int childCount,
        String couponCode
) {
    // ...
}
```

이 코드는 보기 좋지 않습니다. 하지만 필요한 정보가 명확합니다. 반대로 아래처럼 감추면 문제가 덜 보입니다.

```java
void reserve(ReservationRequest request) {
    // request 안에 무엇이 필요한지 호출부에서는 잘 보이지 않는다.
}
```

물론 `ReservationRequest`가 도메인 개념을 제대로 표현한다면 좋은 코드일 수 있습니다. 하지만 단순히 긴 매개변수를 숨기려고 만든 데이터 덩어리라면 스멜을 감춘 것뿐입니다.

## 정리

13장은 "나쁜 코드를 작성하라"는 장이 아닙니다. 정확히는 **나쁜 코드를 좋은 척 숨기지 말라**는 장입니다.

핵심은 세 가지입니다.

1. 나쁜 코드는 팀에 보내는 신호가 될 수 있다.
2. 신호를 남길 때도 정보는 보존하고, 다음 리팩터링은 쉬워져야 한다.
3. 문제는 한눈에 보일 만큼 명확해야 한다.

좋은 코드는 읽기 쉽고 변경하기 쉬운 코드입니다. 하지만 당장 거기까지 갈 수 없다면, 적어도 다음 사람이 어디를 고쳐야 할지 알 수 있게 남겨야 합니다.

그것이 이 장에서 말하는 안티-리팩터링의 목적입니다.
