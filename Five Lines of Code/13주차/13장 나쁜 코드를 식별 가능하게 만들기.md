# 13장 나쁜 코드를 식별 가능하게 만들기

## 핵심 주제

나쁜 코드를 완전히 리팩터링할 수 없는 상황에서는 문제를 어설프게 숨기기보다, 나쁜 코드임을 명확히 드러내야 한다. 이를 책에서는 **안티-리팩터링(anti-refactoring)** 이라고 설명한다.

안티-리팩터링의 목적은 코드를 망치는 것이 아니라, 리팩터링이 필요한 부분을 식별 가능하게 만들어 이후 개선을 쉽게 하는 것이다.

## 나쁜 코드를 드러내야 하는 이유

시간 부족, 우선순위, 복잡한 기존 구조 때문에 코드를 원하는 수준까지 리팩터링하지 못할 때가 있다. 이때 코드를 "나쁘지 않아 보이게" 정리하면 문제가 숨겨진다.

나쁜 코드를 명확히 남기면 다음과 같은 효과가 있다.

- 리팩터링 대상이 다시 발견되기 쉽다.
- 현재 개발 방식이나 일정이 지속 가능하지 않다는 신호가 된다.
- 깨끗한 코드와 레거시 코드를 구분할 수 있다.
- 코드 품질 상태를 팀이 더 정확히 인식할 수 있다.

단, 나쁜 코드를 신호로 남기려면 팀의 심리적 안정감이 필요하다. 문제를 드러낸 사람이 비난받는 문화에서는 개발자가 나쁜 코드를 숨기게 된다.

## 깨끗한 코드와 레거시 코드의 분리

코드베이스에서 깨끗한 코드와 레거시 코드가 구분되면 리팩터링 우선순위를 잡기 쉽다. 책은 애매하게 "나쁘지 않은 코드"보다 명확히 나쁜 코드가 더 낫다고 본다.

이유는 다음과 같다.

- 애매한 코드는 리팩터링 필요성을 숨긴다.
- 명확히 나쁜 코드는 개발자의 주의를 끈다.
- 깨끗한 파일은 깨끗한 상태를 더 오래 유지한다.
- 나쁜 코드가 방치된 파일은 주변에 더 많은 나쁜 코드가 추가되기 쉽다.

이는 깨진 유리창 이론과 연결된다. 이미 망가진 곳에는 더 쉽게 나쁜 변경이 쌓이고, 깨끗한 곳은 더 조심해서 다루게 된다.

## 나쁜 코드를 찾는 방법

나쁜 코드를 완벽하게 판별하는 기준은 없다. 가독성과 설계 품질은 어느 정도 주관적이기 때문이다. 책에서는 나쁜 코드를 식별하기 위한 대표적인 기준으로 세 가지를 소개한다.

### 1. 이 책의 규칙

`Five Lines of Code`의 규칙은 코드 스멜보다 구체적이고 찾기 쉽다. 대표 규칙은 다음과 같다.

- 메서드는 다섯 줄 이내로 유지한다.
- `if`는 함수의 시작에만 둔다.
- `else`를 사용하지 않는다.
- 타입 코드를 클래스로 바꾼다.
- 공통 접사를 사용하지 않는다.

이 규칙들은 보편적인 진리라기보다 리팩터링 감각을 훈련하기 위한 도구다. 장점은 경험이 부족해도 위반 여부를 비교적 쉽게 발견할 수 있다는 점이다.

### 2. 코드 스멜

코드 스멜(code smell)은 코드에 문제가 있을 가능성을 나타내는 증상이다. 대표적인 예는 다음과 같다.

- 중복 코드
- 긴 메서드
- 긴 매개변수 목록
- 매직 넘버
- 타입 코드
- 데이터 클래스

코드 스멜은 많은 리팩터링 서적에서 사용하는 기준이지만, 실제로 잘 활용하려면 경험과 훈련이 필요하다. 일부 스멜은 쉽게 보이지만, 일부는 맥락에 따라 해석이 달라질 수 있다.

### 3. 복잡도 지표

복잡도 지표는 코드 품질을 계산 가능한 값으로 추정하는 방법이다.

**순환 복잡도(cyclomatic complexity)** 는 코드를 통과할 수 있는 경로의 수를 계산한다. `if`, `for`, `while`, `&&`, `||` 등이 많을수록 복잡도가 높아진다. 경로가 많다는 것은 테스트해야 할 경우의 수도 많다는 뜻이다.

**인지 복잡도(cognitive complexity)** 는 사람이 코드를 읽을 때 머릿속에 유지해야 하는 정보의 양을 추정한다. 특히 중첩이 깊을수록 이해 비용이 커진다.

순환 복잡도는 객관적인 흐름 수에 가깝고, 인지 복잡도는 사람이 느끼는 읽기 어려움에 더 가깝다.

## 나쁜 코드를 안전하게 드러내는 세 가지 규칙

나쁜 코드를 눈에 띄게 만들 때는 다음 세 가지 규칙을 지켜야 한다.

### 1. 올바른 정보를 훼손하지 말 것

기존 코드가 가지고 있는 정확한 정보는 보존해야 한다. 메서드 이름, 변수 이름, 타입, 정확한 주석이 의미 있는 정보를 담고 있다면 나쁘게 보이게 하려고 제거하거나 왜곡하면 안 된다.

제거할 수 있는 것은 오래되었거나 부정확하거나 불필요한 정보다.

### 2. 향후 리팩터링을 어렵게 만들지 말 것

나쁜 코드를 드러내는 목적은 이후 리팩터링을 쉽게 하기 위함이다. 따라서 다음 사람이 코드를 고칠 때 더 많은 정보를 얻을 수 있어야 한다.

메서드를 추출할 위치에 빈 줄을 넣거나, 캡슐화할 데이터를 이름으로 묶어두는 것은 이후 리팩터링 방향을 보여준다.

### 3. 결과를 한눈에 알아볼 수 있게 할 것

표시는 명확해야 한다. 코드가 신호로 기능하려면 개발자가 빠르게 알아차릴 수 있어야 한다.

좋은 표식은 다음 조건을 만족한다.

- 문제 위치가 눈에 띈다.
- 리팩터링 방향을 추정할 수 있다.
- 쉽게 되돌릴 수 있다.
- 기존 정보가 사라지지 않는다.

## 나쁜 코드를 눈에 띄게 만드는 방법

책에서 소개하는 방법들은 최종 설계가 아니라 리팩터링 대상을 드러내는 표식이다.

### 열거형 사용

부울이나 정수 타입 코드 대신 열거형을 사용하면 리팩터링 대상이 더 잘 보인다.

열거형의 장점은 다음과 같다.

- 타입 코드가 명확히 드러난다.
- 가능한 값에 이름이 붙어 의미가 보존된다.
- 이후 클래스로 타입 코드를 대체하기 쉽다.
- 분기 로직을 각 타입의 클래스로 옮기는 방향이 보인다.

열거형은 완전한 해결책은 아니지만, 타입 코드가 숨어 있는 상태보다 리팩터링 방향이 명확하다.

### 정수와 문자열을 타입 코드로 사용

빠른 실험이나 검증 단계에서는 정수나 문자열을 타입 코드로 사용할 수 있다. 이 방식은 좋은 최종 구조는 아니지만, 타입 코드가 존재한다는 사실을 눈에 띄게 만든다.

주의할 점은 값의 의미를 잃지 않아야 한다는 것이다. 문자열이나 상수 이름이 의미를 담고 있다면 이후 열거형으로 전환하기 쉽다.

### 매직 넘버 사용

매직 넘버는 대부분의 개발자가 쉽게 알아차리는 코드 스멜이다. 따라서 리팩터링 필요성을 드러내는 표식이 될 수 있다.

다만 매직 넘버는 정보를 훼손할 위험이 크다. 기존 상수 이름이 정확한 정보를 담고 있다면 인라인 값으로 바꾸면 안 된다. 값의 의미가 불분명하다면 주석을 통해 맥락을 보존해야 한다.

### 주석 사용

주석은 일반적으로 줄이는 것이 좋지만, 나쁜 코드를 드러내는 상황에서는 정보를 보존하고 리팩터링 위치를 표시하는 도구가 될 수 있다.

특히 메서드 이름으로 사용할 수 있는 주석은 향후 메서드 추출의 후보를 알려준다.

주석을 사용할 때의 기준은 다음과 같다.

- 정확한 정보만 적는다.
- 오해를 만들지 않는다.
- 리팩터링 방향을 보여준다.
- 불필요한 설명을 반복하지 않는다.

### 공백 사용

빈 줄은 코드 덩어리를 시각적으로 구분한다. 메서드 추출 위치나 데이터 캡슐화 후보를 표시할 때 유용하다.

공백은 이름을 붙일 만큼 충분히 이해하지 못했지만 구조적 덩어리는 보일 때 사용할 수 있다.

활용 예시는 다음과 같다.

- 관련 문장끼리 묶기
- 필드 그룹을 구분하기
- 메서드 추출 후보를 표시하기
- 데이터 캡슐화 후보를 표시하기

### 이름을 기준으로 항목 그룹화

공통 접사는 보통 제거해야 할 냄새지만, 캡슐화할 후보를 드러내는 데는 유용할 수 있다.

여러 필드가 같은 접두사를 공유한다면 하나의 클래스로 묶어야 할 가능성이 있다.

공통 접사는 다음 리팩터링의 신호가 된다.

- 데이터 캡슐화
- 클래스 추출
- 관련 필드와 메서드의 응집도 강화

### 이름에 컨텍스트 추가

필드나 메서드가 같은 개념에 속하지만 이름에서 드러나지 않는 경우, 공통 컨텍스트를 이름에 추가할 수 있다.

이 방식은 향후 캡슐화 대상을 더 쉽게 찾게 한다. 단, 추가하는 컨텍스트는 정확해야 한다. 잘못된 컨텍스트는 이후 리팩터링을 오히려 방해한다.

### 긴 메서드 만들기

잘못 추출된 여러 메서드는 문제를 숨길 수 있다. 이 경우 하나의 긴 메서드로 합쳐 전체 흐름을 다시 드러내는 편이 나을 수 있다.

긴 메서드는 대부분의 개발자가 리팩터링 대상으로 인식한다. 단, 기존 메서드 이름이 의미 있는 정보를 담고 있다면 주석 등으로 보존해야 한다.

### 많은 매개변수 사용

긴 매개변수 목록은 눈에 잘 띄는 코드 스멜이다. 반대로 `Map`, 구조체, 의미 없는 데이터 객체로 매개변수를 감추면 문제가 덜 보인다.

많은 매개변수를 명시하면 다음 정보가 보존된다.

- 필요한 데이터의 이름
- 필요한 데이터의 타입
- 어떤 값들이 함께 사용되는지
- 어떤 값들이 하나의 클래스로 묶일 수 있는지

긴 매개변수 목록은 좋은 최종 구조는 아니지만, 이후 값 객체나 도메인 객체로 묶기 위한 단서를 제공한다.

### getter와 setter 사용

전역 변수나 공개 필드 대신 getter와 setter를 사용하면 데이터 접근 지점이 명확해진다.

이 방식은 캡슐화의 첫 단계로 볼 수 있다. 이후 데이터를 사용하는 코드를 캡슐화된 클래스 내부로 옮기면 getter와 setter는 점차 사라질 수 있다.

getter와 setter는 정의부와 호출부 모두에서 쉽게 검색되므로 리팩터링 대상을 찾기 쉽다.

## 요약

- 안티-리팩터링은 코드를 일부러 망치는 것이 아니라, 나쁜 코드를 식별 가능하게 만드는 활동이다.
- 나쁜 코드를 숨기면 리팩터링 대상이 사라진 것처럼 보인다.
- 깨끗한 코드와 레거시 코드를 구분하면 리팩터링 우선순위를 잡기 쉽다.
- 나쁜 코드를 찾는 기준에는 이 책의 규칙, 코드 스멜, 순환 복잡도, 인지 복잡도가 있다.
- 나쁜 코드를 드러낼 때는 정보를 훼손하지 않아야 한다.
- 나쁜 코드를 드러내는 작업은 이후 리팩터링을 더 쉽게 해야 한다.
- 표시된 문제는 한눈에 알아볼 수 있어야 한다.
- 열거형, 타입 코드, 매직 넘버, 주석, 공백, 공통 접사, 긴 메서드, 긴 매개변수 목록, getter/setter는 리팩터링 대상을 드러내는 표식이 될 수 있다.
- 최종 목표는 나쁜 코드를 유지하는 것이 아니라, 문제를 숨기지 않고 다음 개선으로 연결하는 것이다.
