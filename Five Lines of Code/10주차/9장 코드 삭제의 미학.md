# 9장 코드 삭제의 미학

9장의 핵심은 명확합니다. **코드는 자산이면서 동시에 비용**입니다.

코드가 많아질수록 읽어야 할 것, 테스트해야 할 것, 수정해야 할 것, 의심해야 할 것이 늘어납니다. 그래서 좋은 개발은 코드를 잘 추가하는 일만이 아니라, 더 이상 도움이 되지 않는 코드를 정확히 지우는 일까지 포함합니다.

이번 장에서 다룰 내용
- 코드가 개발 속도를 어떻게 늦추는지 이해하기
- 우발적인 낭비를 방지하기 위한 한계 설정하기
- 스트랭글러 무화과 패턴으로 전환 처리하기
- 스파이크 및 스태빌라이즈 패턴으로 낭비 최소화하기
- 도움이 되지 않는 모든 것을 제거하기

---

# 다음 시대는 코드를 지우는 시대일 것이다.

소프트웨어는 시간이 지날수록 자연스럽게 커집니다. 기능이 추가되고, 예외가 늘고, 임시 코드가 남고, 설정과 테스트와 문서도 함께 쌓입니다.

문제는 코드가 많아질수록 개발 속도가 선형으로 느려지지 않는다는 점입니다. 코드 사이의 관계가 늘어나기 때문에, 작은 변경도 더 많은 맥락을 요구합니다.

그래서 코드 삭제는 단순한 정리가 아닙니다.

- 이해해야 할 범위를 줄입니다.
- 변경의 위험을 줄입니다.
- 테스트와 배포 비용을 줄입니다.
- 팀이 실제로 중요한 코드에 집중하게 합니다.

좋은 코드베이스는 많은 것을 담고 있는 코드베이스가 아니라, **필요한 것만 남아 있는 코드베이스**입니다.

---

# 복잡성을 제거하기 위한 코드 삭제

책은 개발을 느리게 만드는 원인을 네 가지로 나눕니다.

### 경험 부족으로 인한 기술적 무지

기술적 무지는 경험 부족에서 옵니다. 더 나은 방법을 몰라서 복잡한 구조를 만들거나, 언어와 도구의 기능을 제대로 활용하지 못하는 경우입니다.

이 문제는 코드 삭제만으로 해결되지 않습니다. 학습, 리뷰, 페어 프로그래밍, 작은 리팩터링 경험이 필요합니다.

### 시간 압박으로 인한 기술적 낭비

기술적 낭비는 당장 필요하지 않은 코드가 남는 경우입니다.

예를 들어 언젠가 쓸 것 같아서 만든 추상화, 실제 사용자가 없는 설정, 한 번 쓰고 버려야 했던 임시 코드가 여기에 속합니다.

기술적 낭비는 이자를 내는 부채도 아닙니다. 가치 없이 비용만 만듭니다. 발견하면 삭제하는 편이 맞습니다.

### 환경에 따른 기술적 부채

기술적 부채는 상황 때문에 의도적으로 감수한 비용입니다.

마감, 시장 검증, 외부 제약 때문에 완벽하지 않은 선택을 할 수 있습니다. 다만 부채는 갚을 계획이 있을 때만 부채입니다. 계획 없이 방치되면 낭비나 장애물이 됩니다.

### 성장에 따른 기술적 장애물

기술적 장애물은 코드베이스가 커지면서 자연스럽게 생기는 복잡성입니다.

비즈니스가 복잡하면 소프트웨어도 어느 정도 복잡할 수밖에 없습니다. 이 복잡성은 완전히 제거할 수 없지만, 더 이상 필요 없는 기능과 경로를 삭제하면서 관리할 수 있습니다.

---

# 친밀도에 따른 코드 분류

모든 코드를 같은 기준으로 다루면 삭제하기 어렵습니다. 책은 코드와의 친밀도, 즉 얼마나 자주 보고 얼마나 잘 아는지에 따라 다르게 접근하라고 말합니다.

- 자주 수정하는 코드는 적극적으로 단순화합니다.
- 가끔 보는 코드는 변경 전후의 검증 수단을 먼저 확보합니다.
- 거의 모르는 코드는 바로 지우기보다 사용 여부를 추적합니다.

삭제의 핵심은 용기가 아니라 근거입니다. 코드가 호출되지 않는지, 기능 플래그가 꺼져 있는지, 로그상 사용자가 없는지 확인한 뒤 삭제해야 합니다.

---

# 레거시 시스템에서의 코드 삭제

레거시 시스템은 한 번에 갈아엎기 어렵습니다. 그래서 책은 큰 교체보다 점진적 전환을 권합니다.

### 스트랭글러 무화과나무 패턴

스트랭글러 무화과나무 패턴은 기존 시스템 옆에 새 시스템을 만들고, 트래픽이나 기능을 조금씩 새 구조로 옮기는 방식입니다.

핵심은 "한 번에 교체"가 아니라 "사용 중인 경로를 점진적으로 줄이기"입니다.

### 코드 개선을 위한 스트랭글러 무화과나무 패턴 사용

이 패턴은 시스템 교체뿐 아니라 코드 개선에도 쓸 수 있습니다.

1. 기존 코드의 경계에 래퍼나 인터페이스를 둡니다.
2. 새 구현을 같은 경계 뒤에 추가합니다.
3. 호출 지점을 조금씩 새 구현으로 옮깁니다.
4. 더 이상 호출되지 않는 기존 구현을 삭제합니다.

이 방식의 장점은 전환 기간 동안 시스템을 계속 동작시킬 수 있다는 점입니다. 삭제는 마지막에 일어나지만, 삭제할 수 있게 만드는 작업은 처음부터 시작됩니다.

# 동결된 프로젝트에서 코드 삭제

동결된 프로젝트는 더 이상 적극적으로 개발하지 않지만, 완전히 버리기도 애매한 프로젝트입니다. 이런 프로젝트가 많아지면 유지보수 비용이 조용히 늘어납니다.

### 바람직한 결과를 기본값으로 설정

동결된 프로젝트는 "유지"가 기본값이면 계속 남습니다. 책은 오히려 **삭제를 기본값**으로 두라고 말합니다.

살려야 한다면 이유가 있어야 합니다.

- 실제 사용자가 있습니다.
- 법적 또는 운영상 보관이 필요합니다.
- 다른 시스템이 여전히 의존합니다.
- 가까운 시점에 다시 활성화될 계획이 있습니다.

이유가 없다면 보존이 아니라 방치입니다.

### 스파이크와 스태빌라이즈로 낭비 줄이기

스파이크는 빠르게 실험하기 위한 코드입니다. 스태빌라이즈는 실험에서 살아남은 아이디어를 제품 수준으로 다듬는 과정입니다.

중요한 점은 스파이크 코드를 그대로 제품 코드로 승격하지 않는 것입니다.

- 스파이크는 배움을 얻기 위한 코드입니다.
- 스태빌라이즈는 유지할 코드를 다시 쓰는 단계입니다.
- 실험이 끝나면 스파이크는 삭제합니다.

이 구분이 없으면 실험용 코드가 영구 코드가 됩니다.

# 버전 관리에서 브랜치 삭제

브랜치도 코드처럼 비용을 만듭니다.

오래된 브랜치가 많으면 무엇이 살아 있는 작업인지, 어떤 변경이 아직 필요한지 계속 판단해야 합니다. 브랜치가 많을수록 머지 충돌과 맥락 전환도 늘어납니다.

### 브랜치 제한으로 낭비 최소화

브랜치는 짧게 유지하고, 오래된 브랜치는 닫거나 삭제해야 합니다.

실무 기준은 단순할수록 좋습니다.

- 머지된 브랜치는 삭제합니다.
- 일정 기간 업데이트가 없는 브랜치는 소유자를 확인합니다.
- 소유자와 목적이 없는 브랜치는 삭제 후보로 둡니다.

브랜치 삭제는 저장 공간 문제가 아니라 인지 부하 문제입니다.

---

# 코드 문서 삭제

문서도 오래되면 코드만큼 위험합니다.

문서가 유용하려면 세 가지 조건을 만족해야 합니다.

- 관련성이 있어야 합니다.
- 정확해야 합니다.
- 찾을 수 있어야 합니다.

셋 중 하나라도 빠지면 문서는 도움보다 혼란을 줍니다.

### 지식을 문서화하는 방법을 결정하는 알고리즘

어떤 지식을 발견했을 때 바로 문서부터 쓰지 말고, 다음 순서로 판단합니다.

1. 코드 자체로 표현할 수 있는가?
2. 테스트로 고정할 수 있는가?
3. 자동화나 도구로 강제할 수 있는가?
4. 그래도 어렵다면 문서로 남길 가치가 있는가?

문서는 마지막 수단이어야 합니다. 사람이 읽고 기억해야 하는 지식보다, 시스템이 자동으로 강제하는 지식이 더 안전합니다.

---

# 테스트 코드 삭제

테스트는 많을수록 좋은 것이 아닙니다. **신뢰할 수 있는 테스트가 많을수록 좋은 것**입니다.

잘못된 테스트는 개발자를 느리게 만들고, 테스트 실패에 둔감하게 만듭니다.

### 낙관적 테스트 삭제

낙관적 테스트는 실제로 실패해야 할 상황을 제대로 검증하지 못하는 테스트입니다.

예를 들어 결과를 대충 확인하거나, 중요한 assertion이 빠져 있거나, 구현을 호출만 하고 끝나는 테스트가 여기에 속합니다. 이런 테스트는 안정감을 주지만 실제 결함을 잡지 못합니다.

### 비관적 테스트 삭제

비관적 테스트는 너무 많은 것을 의심해서 사소한 내부 구현 변화에도 실패하는 테스트입니다.

이런 테스트는 리팩터링을 방해합니다. 테스트는 외부 동작을 지켜야지, 불필요한 내부 구조까지 고정하면 안 됩니다.

### 불안정 테스트 수정 또는 삭제

불안정 테스트는 같은 코드에서도 통과와 실패가 반복되는 테스트입니다.

원인이 시간, 네트워크, 랜덤 값, 실행 순서라면 먼저 고쳐야 합니다. 고칠 수 없고 신뢰도만 떨어뜨린다면 삭제하는 편이 낫습니다.

### 복잡한 테스트를 제거하기 위한 코드 리팩터링

테스트가 지나치게 복잡하다면 테스트만의 문제가 아닐 수 있습니다. 테스트 대상 코드가 너무 많은 책임을 가지고 있거나, 의존성이 강하게 얽혀 있을 가능성이 큽니다.

이 경우 테스트를 억지로 유지하기보다 코드를 리팩터링해서 테스트가 단순해지게 만들어야 합니다.

### 속도를 높이는 테스트 문화

느린 테스트는 결국 덜 실행됩니다.

테스트 셋은 빠른 피드백을 주는 방향으로 관리해야 합니다. 느린 통합 테스트와 빠른 단위 테스트를 구분하고, 자주 실행해야 하는 테스트는 작고 빠르게 유지해야 합니다.

# 설정 코드 삭제

설정은 유연성을 주지만, 동시에 경우의 수를 늘립니다.

설정이 많아질수록 코드는 "어떤 값으로 실행되는가"에 따라 다른 프로그램처럼 동작합니다. 이는 테스트와 디버깅 비용을 크게 늘립니다.

### 설정의 예상 수명으로 범위 지정

설정을 만들 때는 예상 수명을 정해야 합니다.

- 임시 전환용 설정인지
- 실험용 기능 플래그인지
- 환경별로 계속 필요한 설정인지

임시 설정이라면 삭제 날짜나 삭제 조건을 함께 정해야 합니다. 그렇지 않으면 설정은 영구 복잡성이 됩니다.

# 라이브러리 제거를 위한 코드 삭제

라이브러리는 직접 구현하지 않아도 되는 장점을 주지만, 의존성 비용도 만듭니다.

- 업데이트를 따라가야 합니다.
- 보안 이슈를 확인해야 합니다.
- API 변경에 대응해야 합니다.
- 내부 동작을 완전히 통제하기 어렵습니다.

### 외부 라이브러리에 대한 의존도 제한

라이브러리를 쓰더라도 의존 범위를 좁히는 것이 좋습니다.

특정 라이브러리 API가 코드 전체에 퍼지면 제거가 어렵습니다. 반대로 얇은 래퍼나 경계를 두면 교체나 삭제가 쉬워집니다.

라이브러리를 제거할 때도 기준은 같습니다. 실제 가치보다 유지 비용이 크다면 삭제 후보입니다.

---

# 작동 중인 기능에서 코드 삭제 요약

작동 중인 코드를 삭제할 때는 감으로 지우면 안 됩니다.

핵심 절차는 다음과 같습니다.

1. 사용 여부를 확인합니다.
2. 삭제 범위를 작게 나눕니다.
3. 테스트, 로그, 모니터링으로 검증합니다.
4. 전환 기간이 필요하면 스트랭글러 방식으로 옮깁니다.
5. 더 이상 호출되지 않는 경로를 삭제합니다.

삭제는 마지막 행동이지만, 삭제 가능성을 만드는 설계는 계속 필요합니다.

---

# 요약

- 기술적 무지, 기술적 낭비, 기술적 부채, 기술적 장애물은 개발이 느려지는 원인이 됩니다.
- 기술적 낭비는 가치 없이 비용만 만들기 때문에 발견하면 삭제해야 합니다.
- 기술적 부채는 갚을 계획이 있을 때만 의미가 있습니다.
- 스트랭글러 무화과나무 패턴은 레거시 시스템을 점진적으로 대체하고, 마지막에 기존 코드를 삭제할 수 있게 만듭니다.
- 스파이크와 스태빌라이즈를 구분하면 실험용 코드가 제품 코드에 섞이는 낭비를 줄일 수 있습니다.
