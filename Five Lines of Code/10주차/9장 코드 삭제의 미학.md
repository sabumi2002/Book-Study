# 9장 코드 삭제의 미학

9장은 책 전체의 마지막 메시지에 가깝습니다. 좋은 코드는 더 많이 작성한 코드가 아니라, **필요한 것만 남아 있는 코드**입니다.

개발자는 보통 코드를 추가하는 일에 익숙합니다. 기능을 만들고, 옵션을 늘리고, 예외를 처리하고, 테스트를 더합니다. 하지만 시간이 지나면 프로젝트를 어렵게 만드는 것은 부족한 코드보다 **너무 많이 남아 있는 코드**인 경우가 많습니다.

이 장은 "삭제"를 단순 정리 작업이 아니라 중요한 설계 활동으로 봅니다.

> 코드를 삭제하는 능력은 코드를 작성하는 능력만큼 중요하다.

삭제는 위험해 보입니다. 하지만 유지되지 않는 코드, 사용되지 않는 기능, 오래된 테스트, 과도한 설정, 불필요한 라이브러리는 계속 비용을 만듭니다. 읽어야 하고, 빌드해야 하고, 테스트해야 하고, 변경 시 함께 고려해야 합니다. 삭제하지 않는 선택도 비용이 있는 선택입니다.

이번 장에서 다룰 내용
- 코드 삭제가 복잡성을 줄이는 이유
- 기술적 무지, 기술적 낭비, 기술적 부채, 기술적 장애물
- 친밀도에 따른 코드 분류
- 레거시 시스템과 스트랭글러 무화과나무 패턴
- 동결된 프로젝트와 스파이크/스태빌라이즈
- 브랜치, 문서, 테스트, 설정, 라이브러리 삭제
- 작동 중인 기능에서 코드를 삭제하는 관점

# 다음 시대는 코드를 지우는 시대일 것이다.

소프트웨어 개발은 오랫동안 "무엇을 만들 것인가"에 집중해 왔습니다. 하지만 시스템이 오래될수록 더 중요한 질문은 바뀝니다.

> 무엇을 더 만들 것인가가 아니라, 무엇을 더 이상 유지하지 않을 것인가?

코드는 작성되는 순간부터 유지보수 비용을 만듭니다. 기능 하나가 추가되면 코드만 늘어나는 것이 아닙니다. 테스트, 문서, 설정, 배포 절차, 장애 대응 지식, 팀원의 이해 비용도 함께 늘어납니다.

그래서 오래된 시스템에서 생산성을 떨어뜨리는 것은 대개 "코드가 부족한 상태"가 아니라 "삭제되지 않은 결정이 너무 많은 상태"입니다.

삭제는 다음 효과를 만듭니다.

- 읽어야 할 코드가 줄어든다.
- 변경 영향 범위가 좁아진다.
- 테스트와 빌드 시간이 줄어든다.
- 죽은 기능을 고려하지 않아도 된다.
- 팀이 현재 중요한 도메인에 집중할 수 있다.

이 장의 관점에서 삭제는 용기가 아니라 시스템 운영 능력입니다. 무엇이 더 이상 필요 없는지 판단하고, 안전하게 제거하고, 제거된 상태를 기본값으로 만드는 능력이 중요합니다.

# 복잡성을 제거하기 위한 코드 삭제

복잡성은 코드가 어렵게 생겼을 때만 생기지 않습니다. 사용되지 않는 코드, 거의 쓰이지 않는 설정, 오래된 분기, 예외적인 고객을 위한 임시 처리도 모두 복잡성입니다.

예를 들어 결제 시스템에 다음 분기가 남아 있다고 해봅시다.

```typescript
if (paymentProvider === "legacy-v1") {
  chargeWithLegacyProvider(order);
} else {
  chargeWithCurrentProvider(order);
}
```

`legacy-v1`을 실제로 아무도 쓰지 않는다면 이 코드는 단순한 조건문이 아닙니다. 개발자는 결제 로직을 바꿀 때마다 두 경로를 모두 고려해야 하고, 테스트도 두 경로를 의식해야 하며, 장애가 나면 오래된 코드도 의심해야 합니다.

삭제는 이런 숨은 비용을 제거합니다.

```typescript
chargeWithCurrentProvider(order);
```

코드가 줄어든 것보다 중요한 것은 **생각해야 할 경우의 수가 줄었다는 점**입니다.

책은 복잡성이 생기는 원인을 몇 가지로 나눕니다. 이 분류는 삭제할 대상을 찾는 데 유용합니다.

### 경험 부족으로 인한 기술적 무지

기술적 무지는 당시에는 더 나은 방법을 몰라서 생긴 코드입니다.

예를 들어 같은 검증 로직이 여러 곳에 복사되어 있거나, 언어와 프레임워크가 제공하는 표준 기능을 모르고 직접 구현한 경우가 여기에 해당합니다.

```typescript
function isEmpty(text: string | null | undefined) {
  return text === null || text === undefined || text.length === 0;
}
```

이 코드가 문제라는 뜻은 아닙니다. 다만 프로젝트가 성장하고 팀의 지식이 늘어나면, 더 단순한 표준 방식이나 공통 유틸리티로 대체할 수 있습니다. 이때 과거의 코드를 계속 유지하는 것은 학습 비용을 고정시키는 일입니다.

기술적 무지는 비난할 대상이 아닙니다. 당시에는 최선이었을 수 있습니다. 하지만 지금 더 나은 방법을 알고 있다면, 오래된 구현을 제거해야 합니다.

### 시간 압박으로 인한 기술적 낭비

기술적 낭비는 급하게 만들었지만 더 이상 가치가 없어진 코드입니다.

대표적인 예는 임시 기능, 일회성 마이그레이션 코드, 실험용 플래그, 출시 직전의 우회 처리입니다.

```typescript
if (campaignId === "spring-2024-hotfix") {
  applyTemporaryDiscount(cart);
}
```

문제는 임시 코드가 임시로 남지 않는다는 점입니다. 출시가 끝나도 삭제되지 않으면 이후 모든 개발자는 이 조건을 이해해야 합니다. 시간이 지나면 아무도 왜 있는지 모르는 코드가 됩니다.

기술적 낭비를 줄이려면 임시 코드를 만들 때부터 삭제 조건을 함께 정해야 합니다.

- 언제 제거할 것인가?
- 어떤 지표가 0이 되면 삭제할 것인가?
- 어떤 릴리스 이후에 플래그를 제거할 것인가?
- 삭제 담당자는 누구인가?

삭제 계획이 없는 임시는 사실상 영구 코드입니다.

### 환경에 따른 기술적 부채

기술적 부채는 당시에는 합리적인 선택이었지만, 시간이 지나며 비용이 된 코드입니다.

예를 들어 초기에는 직접 만든 인증 로직이 빠르고 단순했을 수 있습니다. 하지만 조직이 커지고 보안 요구가 높아지면 그 코드는 위험과 비용이 됩니다.

기술적 부채의 특징은 "처음부터 틀린 코드"가 아니라는 점입니다. 환경이 바뀌면서 더 이상 맞지 않게 된 코드입니다.

따라서 기술적 부채를 다룰 때는 감정적으로 접근하면 안 됩니다.

- 당시 제약에서는 맞는 선택이었는가?
- 지금의 요구와 규모에서도 여전히 맞는가?
- 대체하면 어떤 비용이 줄어드는가?
- 한 번에 제거할 수 있는가, 점진적으로 대체해야 하는가?

삭제는 과거를 부정하는 일이 아니라 현재의 시스템에 맞게 책임을 조정하는 일입니다.

### 성장에 따른 기술적 장애물

기술적 장애물은 이제 팀의 성장을 직접 막는 코드입니다. 단순히 지저분한 정도를 넘어, 새 기능 개발을 느리게 만들고 장애 가능성을 높이며 팀의 의사결정을 방해합니다.

예를 들어 모든 비즈니스 규칙이 하나의 거대한 서비스 클래스에 모여 있다면, 새 기능을 추가할 때마다 충돌이 생기고 테스트가 어려워집니다. 이 경우 일부 코드를 삭제하고 책임을 나누는 일은 선택이 아니라 성장 조건입니다.

장애물 수준의 코드는 "언젠가 정리"로는 해결되지 않습니다. 기능 개발의 일부로 작게라도 제거해야 합니다.

# 친밀도에 따른 코드 분류

삭제를 잘하려면 코드와의 관계를 구분해야 합니다. 책은 코드를 친밀도 관점에서 볼 수 있게 합니다.

실무적으로는 다음처럼 나눠볼 수 있습니다.

- 자주 읽고 자주 바꾸는 코드
- 가끔 읽지만 거의 바꾸지 않는 코드
- 존재는 알지만 아무도 자세히 모르는 코드
- 사용 여부조차 불확실한 코드

삭제 우선순위는 마지막 두 범주에서 시작하는 것이 좋습니다. 아무도 모르는 코드를 바로 삭제하라는 뜻은 아닙니다. 먼저 관찰하고, 사용 여부를 확인하고, 경계를 좁혀야 합니다.

코드 친밀도가 낮을수록 다음 전략이 필요합니다.

- 로그나 메트릭으로 실제 사용 여부를 확인한다.
- 기능 플래그로 트래픽을 차단해 본다.
- 호출 지점을 검색한다.
- 테스트와 배포 이력을 확인한다.
- 작은 단위로 삭제하고 검증한다.

친밀도는 감정이 아니라 정보의 양입니다. 잘 아는 코드는 자신 있게 바꿀 수 있고, 잘 모르는 코드는 먼저 관찰해야 합니다.

# 레거시 시스템에서의 코드 삭제

레거시 시스템에서 삭제는 특히 어렵습니다. 코드가 오래되었고, 테스트가 부족하며, 실제 사용자가 어디에 의존하는지 불분명하기 때문입니다.

이때 한 번에 갈아엎는 방식은 위험합니다. 책이 제시하는 중요한 접근은 **스트랭글러 무화과나무 패턴**입니다.

### 스트랭글러 무화과나무 패턴

스트랭글러 무화과나무 패턴은 기존 시스템을 한 번에 교체하지 않고, 새 시스템이 조금씩 기존 시스템의 역할을 대체하게 만드는 방식입니다.

비유적으로는 새 나무가 오래된 나무를 감싸며 자라다가 결국 오래된 나무를 대체하는 모습입니다.

소프트웨어에서는 보통 다음 순서로 진행됩니다.

1. 기존 시스템 앞에 경계를 둔다.
2. 새 기능은 새 구조로 구현한다.
3. 기존 기능 일부를 새 구조로 옮긴다.
4. 트래픽이나 호출을 점진적으로 새 구조로 보낸다.
5. 더 이상 호출되지 않는 기존 코드를 삭제한다.

이 방식의 장점은 위험을 나눌 수 있다는 점입니다. 전체를 한 번에 바꾸지 않고, 사용 중인 기능과 사용하지 않는 기능을 분리해 가며 삭제할 수 있습니다.

### 코드 개선을 위한 스트랭글러 무화과나무 패턴 사용

스트랭글러 패턴은 대규모 시스템 교체에만 쓰는 것이 아닙니다. 작은 코드 개선에도 사용할 수 있습니다.

예를 들어 오래된 할인 계산 로직이 복잡한 함수 하나에 들어 있다고 해봅시다.

```typescript
function calculateDiscount(order: Order) {
  if (order.customerType === "vip") {
    // ...
  }
  if (order.couponCode) {
    // ...
  }
  if (order.eventId) {
    // ...
  }
}
```

이 함수를 한 번에 새로 쓰기 어렵다면, 새 할인 규칙부터 별도 객체로 옮길 수 있습니다.

```typescript
function calculateDiscount(order: Order) {
  if (newDiscountPolicyApplies(order)) {
    return newDiscountPolicy.calculate(order);
  }

  return calculateLegacyDiscount(order);
}
```

그다음 기존 규칙을 하나씩 새 정책으로 옮깁니다. 마지막에 `calculateLegacyDiscount()`가 더 이상 필요 없어지면 삭제합니다.

핵심은 삭제를 마지막 이벤트로 보지 않는 것입니다. 삭제는 점진적 대체 과정의 결과입니다.

# 동결된 프로젝트에서 코드 삭제

동결된 프로젝트는 새 기능 개발이 거의 없거나, 유지보수만 하는 상태의 프로젝트입니다. 이런 프로젝트에서는 "건드리지 않는 것이 안전하다"는 생각이 강해집니다.

하지만 동결된 프로젝트에도 비용은 남아 있습니다.

- 보안 패치가 필요하다.
- 빌드 환경이 낡는다.
- 운영자가 코드를 이해해야 한다.
- 장애가 나면 원인을 찾아야 한다.
- 문서와 실제 동작이 어긋난다.

따라서 동결된 프로젝트에서도 삭제는 필요합니다. 다만 공격적으로 리팩터링하기보다, **바람직한 상태를 기본값으로 만들고 낭비를 줄이는 방식**이 적절합니다.

### 바람직한 결과를 기본값으로 설정

사용하지 않는 옵션이 많을수록 시스템은 복잡해집니다. 동결된 프로젝트에서는 더더욱 실제 운영에 필요한 경로만 남기는 것이 좋습니다.

예를 들어 설정 파일에 오래된 모드가 여러 개 남아 있다면, 현재 운영 모드를 기본값으로 고정하고 나머지를 제거할 수 있습니다.

```typescript
// Before
startServer({ mode: config.mode ?? "legacy" });

// After
startServer({ mode: "current" });
```

물론 실제 삭제 전에는 사용 여부를 확인해야 합니다. 하지만 확인이 끝났다면 설정 가능성을 계속 남겨둘 이유는 없습니다. 필요 없는 유연성은 유지보수 비용입니다.

### 스파이크와 스태빌라이즈(안정화)로 낭비 줄이기

스파이크는 빠르게 실험하는 코드입니다. 목적은 정답 구현이 아니라 학습입니다. 문제는 스파이크 코드가 그대로 제품 코드가 되는 순간 생깁니다.

책의 관점에서 좋은 흐름은 다음과 같습니다.

1. 스파이크로 빠르게 배운다.
2. 배운 내용을 바탕으로 안정화된 구현을 다시 만든다.
3. 스파이크 코드는 삭제한다.

스파이크의 가치는 코드 자체가 아니라 학습 결과입니다. 학습이 끝난 뒤에도 스파이크 코드를 계속 유지하면, 불안정한 가정과 급한 구현이 제품의 일부가 됩니다.

# 버전 관리에서 브랜치 삭제

브랜치도 코드처럼 비용을 만듭니다. 오래된 브랜치가 많으면 어떤 작업이 살아 있는지, 어떤 변경이 폐기된 것인지 판단하기 어려워집니다.

특히 장기 브랜치는 다음 문제를 만듭니다.

- main 브랜치와 차이가 커진다.
- 머지 충돌 가능성이 높아진다.
- 리뷰해야 할 변경량이 커진다.
- 완료되지 않은 일이 계속 남아 있는 것처럼 보인다.

브랜치는 작업 단위여야지 저장 창고가 되어서는 안 됩니다.

### 브랜치 제한으로 낭비 최소화

브랜치 낭비를 줄이는 방법은 단순합니다.

- 오래된 브랜치를 주기적으로 삭제한다.
- 병합된 브랜치는 즉시 삭제한다.
- 실험 브랜치는 실험 종료 시 폐기한다.
- 장기 작업은 작은 PR로 나눈다.
- 브랜치 이름과 목적을 명확히 한다.

브랜치를 줄인다는 것은 히스토리를 잃는 일이 아닙니다. Git은 이미 커밋과 병합 기록을 보관합니다. 삭제해야 하는 것은 정보가 아니라 **현재 진행 중인 일처럼 보이는 착시**입니다.

---

# 코드 문서 삭제

문서도 코드처럼 낡습니다. 특히 코드와 같은 내용을 중복해서 설명하는 문서는 빠르게 틀립니다.

예를 들어 README에 API 파라미터 목록이 길게 적혀 있고, 실제 타입 정의가 따로 있다면 둘은 언젠가 어긋납니다. 이때 문서를 더 열심히 고치는 것보다, 문서가 담당해야 할 지식을 다시 정하는 편이 낫습니다.

### 지식을 문서화하는 방법을 결정하는 알고리즘

실무적으로는 다음 기준을 사용할 수 있습니다.

1. 컴파일러나 타입으로 표현할 수 있는가?
   - 가능하면 코드로 표현한다.
2. 테스트로 보장할 수 있는가?
   - 가능하면 테스트로 남긴다.
3. 코드 가까이에 있어야 하는 제약인가?
   - 가능하면 주석으로 남긴다.
4. 사용 방법이나 운영 절차인가?
   - 문서로 남긴다.
5. 더 이상 현재 사실이 아닌가?
   - 삭제한다.

문서 삭제의 목적은 설명을 줄이는 것이 아닙니다. 지식을 가장 잘 유지될 수 있는 위치로 옮기는 것입니다.

좋은 문서는 코드가 말하지 못하는 내용을 다룹니다.

- 왜 이런 결정을 했는가?
- 어떤 운영 절차를 따라야 하는가?
- 어떤 외부 제약이 있는가?
- 새 팀원이 어디서 시작해야 하는가?

반대로 코드에서 자동으로 확인 가능한 정보는 문서보다 코드와 테스트에 두는 편이 안전합니다.

---

# 테스트 코드 삭제

테스트는 많을수록 좋다고 생각하기 쉽습니다. 하지만 테스트도 유지보수 대상입니다. 느리고, 불안정하고, 중복되고, 의미 없는 테스트는 개발 속도를 떨어뜨립니다.

테스트 삭제는 조심해야 합니다. 테스트가 실패하지 않는다고 가치가 없는 것은 아니고, 오래된 테스트가 중요한 회귀를 막고 있을 수도 있습니다. 하지만 모든 테스트를 영원히 보존하는 것도 답은 아닙니다.

### 낙관적 테스트 삭제

낙관적 테스트는 실패 가능성이 거의 없거나, 구현을 그대로 반복하는 테스트입니다.

```typescript
test("adds two numbers", () => {
  expect(add(1, 2)).toBe(3);
});
```

이런 테스트가 항상 나쁘다는 뜻은 아닙니다. 하지만 도메인 의미 없이 단순 언어 기능이나 구현 세부사항만 확인한다면 유지 비용에 비해 가치가 낮습니다.

낙관적 테스트는 다음 질문으로 판단합니다.

- 이 테스트가 실패하면 실제 버그를 잡는가?
- 구현을 바꿔도 요구사항이 같으면 유지되어야 하는가?
- 같은 내용을 더 중요한 테스트가 이미 검증하는가?

답이 모두 약하다면 삭제 후보입니다.

### 비관적 테스트 삭제

비관적 테스트는 너무 많은 내부 동작을 의심해서 구현 변경을 방해하는 테스트입니다.

```typescript
expect(repository.save).toHaveBeenCalledTimes(1);
expect(repository.save).toHaveBeenCalledWith({
  id: "1",
  status: "READY"
});
```

이런 테스트가 외부 동작보다 내부 호출 순서와 횟수에 집착하면, 리팩터링할 때마다 깨집니다. 실제 사용자 관점의 동작은 그대로인데 테스트만 실패한다면 테스트가 설계를 고정하고 있는 것입니다.

비관적 테스트는 삭제하거나 더 높은 수준의 행동 검증으로 바꾸는 편이 좋습니다.

### 불안정 테스트 수정 또는 삭제

불안정한 테스트는 가끔 실패하는 테스트입니다. 이런 테스트는 팀에 큰 피해를 줍니다.

- 실패를 무시하는 문화가 생긴다.
- CI 신뢰도가 떨어진다.
- 실제 회귀와 환경 문제를 구분하기 어렵다.
- 배포 속도가 느려진다.

불안정 테스트는 방치하면 안 됩니다. 원인을 찾고 수정해야 합니다. 시간, 네트워크, 랜덤 값, 외부 서비스 의존성, 테스트 순서 의존성이 흔한 원인입니다.

수정할 수 없다면 임시로 격리하거나 삭제해야 합니다. 실패해도 믿지 않는 테스트는 보호 장치가 아니라 소음입니다.

### 복잡한 테스트를 제거하기 위한 코드 리팩터링

테스트가 복잡하다는 것은 종종 제품 코드가 복잡하다는 신호입니다.

테스트를 만들기 위해 거대한 객체 그래프를 준비해야 하거나, 내부 상태를 억지로 바꿔야 하거나, 많은 mock을 연결해야 한다면 설계가 테스트를 어렵게 만들고 있을 수 있습니다.

이때 테스트를 단순히 삭제하기보다 제품 코드를 리팩터링해야 합니다.

- 생성자를 단순하게 만든다.
- 외부 의존성을 경계 밖으로 밀어낸다.
- 순수 함수를 분리한다.
- 도메인 규칙을 작은 객체로 옮긴다.
- 불필요한 설정 가능성을 제거한다.

좋은 설계는 테스트도 단순하게 만듭니다. 테스트 삭제가 목표가 아니라, 테스트가 자연스럽게 작아지는 구조가 목표입니다.

### 속도를 높이는 테스트 문화

테스트 문화의 핵심은 테스트를 많이 갖는 것이 아니라 **믿을 수 있는 빠른 피드백을 갖는 것**입니다.

이를 위해서는 테스트도 정리 대상이 되어야 합니다.

- 느린 테스트는 별도 단계로 분리한다.
- 중복 테스트는 줄인다.
- 실패 원인이 불명확한 테스트는 고친다.
- 구현 세부사항에 묶인 테스트는 행동 중심으로 바꾼다.
- 더 이상 의미 없는 테스트는 삭제한다.

테스트는 변경을 두렵지 않게 만들기 위한 도구입니다. 테스트 자체가 변경을 두렵게 만든다면 다시 설계해야 합니다.

# 설정 코드 삭제

설정은 유연성을 줍니다. 하지만 모든 유연성은 비용을 가집니다. 설정 값이 많을수록 가능한 조합이 늘어나고, 가능한 조합이 늘어날수록 테스트와 이해 비용이 커집니다.

```typescript
createServer({
  useLegacyAuth: true,
  enableNewCheckout: false,
  retryMode: "custom",
  cacheStrategy: "mixed"
});
```

이 설정들이 모두 실제로 필요한지 확인해야 합니다. 한 번 도입된 설정은 쉽게 남습니다. 하지만 대부분의 설정은 수명이 있습니다.

### 설정의 예상 수명으로 범위 지정

설정을 만들 때는 예상 수명을 함께 정해야 합니다.

- 일회성 마이그레이션 설정인가?
- A/B 테스트가 끝나면 제거할 플래그인가?
- 특정 고객 전용 설정인가?
- 장기적으로 제품의 공식 옵션인가?

수명이 짧은 설정은 코드 구조도 짧은 수명에 맞춰야 합니다. 예를 들어 실험 플래그라면 종료일과 삭제 이슈를 함께 만들어야 합니다. 장기 설정이라면 문서, 테스트, 기본값, 운영 절차가 필요합니다.

문제는 단기 설정을 장기 설정처럼 방치할 때 생깁니다. 사용하지 않는 설정은 삭제해야 합니다. 설정 가능성 자체가 복잡성이기 때문입니다.

# 라이브러리 제거를 위한 코드 삭제

외부 라이브러리는 생산성을 높이지만, 동시에 의존성을 만듭니다.

라이브러리를 추가하면 다음 비용이 생깁니다.

- 보안 업데이트를 추적해야 한다.
- 버전 충돌을 관리해야 한다.
- API 변경에 대응해야 한다.
- 번들 크기나 실행 환경에 영향을 준다.
- 팀이 라이브러리의 동작을 이해해야 한다.

라이브러리가 핵심 문제를 해결하고 있다면 이 비용은 정당합니다. 하지만 작은 기능 하나를 위해 큰 라이브러리를 유지하고 있다면 제거를 고려해야 합니다.

### 외부 라이브러리에 대한 의존도 제한

라이브러리를 안전하게 제거하려면 의존성을 좁혀야 합니다. 코드 곳곳에서 직접 호출하면 제거가 어렵습니다. 반대로 얇은 경계 뒤에 숨겨두면 교체나 삭제가 쉬워집니다.

```typescript
// Before: 여러 곳에서 직접 사용
import dayjs from "dayjs";

const label = dayjs(date).format("YYYY-MM-DD");
```

```typescript
// After: 경계를 둔다
function formatDate(date: Date): string {
  return dayjs(date).format("YYYY-MM-DD");
}
```

이렇게 하면 나중에 `dayjs`를 제거하거나 표준 API로 바꿀 때 변경 범위가 줄어듭니다.

라이브러리 삭제는 단번에 할 수도 있지만, 보통은 다음 순서가 안전합니다.

1. 직접 호출 지점을 찾는다.
2. 프로젝트 내부 경계 함수나 어댑터로 감싼다.
3. 새 구현으로 바꾼다.
4. 사용하지 않는 라이브러리를 제거한다.
5. 빌드와 테스트로 확인한다.

의존성은 적을수록 좋다는 단순한 말이 아닙니다. 핵심은 **의존하는 범위를 통제할 수 있어야 한다**는 점입니다.

# 작동 중인 기능에서 코드 삭제 요약

작동 중인 기능의 코드를 삭제하는 일은 불안합니다. 그래서 삭제에는 절차가 필요합니다.

실무적으로는 다음 순서를 사용할 수 있습니다.

1. 삭제 후보를 찾는다.
   - 사용되지 않는 분기, 오래된 설정, 죽은 기능, 중복 테스트, 직접 호출되는 외부 라이브러리를 찾습니다.
2. 실제 사용 여부를 확인한다.
   - 검색, 로그, 메트릭, 배포 이력, 고객 사용 여부를 확인합니다.
3. 삭제 범위를 좁힌다.
   - 한 번에 크게 지우기보다 경계를 만들고 작은 단위로 제거합니다.
4. 기본값을 바꾼다.
   - 더 이상 원하지 않는 경로가 기본으로 실행되지 않게 합니다.
5. 관찰한다.
   - 오류, 로그, 사용자 영향, 테스트 결과를 확인합니다.
6. 완전히 삭제한다.
   - 코드, 테스트, 설정, 문서, 브랜치, 의존성을 함께 정리합니다.

9장의 핵심은 삭제를 마지막 청소로 보지 않는 것입니다. 삭제는 설계를 단순하게 유지하는 지속적인 활동입니다.

이 책 전체의 흐름과 연결하면 메시지는 더 분명해집니다.

- 긴 함수는 나누고, 의미 없는 함수는 인라인 후 삭제한다.
- 분기문은 다형성으로 옮기고, 남은 타입 코드는 삭제한다.
- 데이터 불변속성은 캡슐화하고, 외부에서 조작하던 경로는 삭제한다.
- 주석은 코드로 옮기고, 중복 설명은 삭제한다.
- 더 이상 쓰지 않는 기능, 테스트, 설정, 문서, 라이브러리는 삭제한다.

결국 리팩터링은 코드를 예쁘게 만드는 일이 아닙니다. 시스템이 현재 필요로 하는 구조만 남기는 일입니다.

> 좋은 코드는 많이 쓴 코드가 아니라, 더 이상 필요 없는 것을 계속 지워낸 뒤에도 남아 있는 코드다.

삭제는 손실이 아니라 집중입니다. 그리고 집중된 코드는 더 읽기 쉽고, 더 고치기 쉽고, 더 오래 살아남습니다.

# 요약
- 기술적 무지, 기술적 낭비, 기술적 부채, 기술적 방해물은 개발이 더디고 더 어려줘지는 원인이 됩니다. 기술적 무지는 일반적으로 경험 부족에서 비롯되며 기술적 우수성에 대한 지속적인 투자를 통해서만 해결됩니다. 기술적 낭비는 시간 압박에서 오는 경우가 많지만 아무런 이득도 제공하지 않기 때문에 게으름과 같습니다. 기술적 부채는 환경에서 발생하며 일시적인 한 수용 가
