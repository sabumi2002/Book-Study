# 8장 주석 자제하기

앞선 장들에서 책은 긴 함수, 분기문, 타입 코드, 잘못 노출된 데이터처럼 **코드 구조 자체가 이해를 방해하는 문제**를 다뤘습니다.

8장의 주제는 주석입니다. 주석은 흔히 좋은 습관처럼 여겨지지만, 이 책은 조금 다르게 봅니다. 주석은 코드를 설명하는 도구이기 전에, 종종 **코드가 자기 의도를 충분히 드러내지 못한다는 신호**입니다.

이번 장의 핵심은 "주석을 모두 없애라"가 아닙니다.

> 주석으로 설명해야 하는 코드는 먼저 코드 자체를 고칠 수 없는지 의심하라.

좋은 이름, 작은 함수, 명확한 타입, 지역화된 불변속성으로 표현할 수 있는 내용이라면 주석보다 코드로 드러내는 편이 낫습니다. 반대로 코드만으로 표현하기 어려운 의도, 제약, 프로세스, 불변속성은 주석으로 남겨야 합니다.

이번 장에서 다룰 내용
- 오래된 주석 제거
- 주석 처리된 코드 제거
- 불필요한 주석 제거
- 메서드 이름으로 주석 대체
- 계획을 위한 주석 사용
- 불변속성을 문서화한 주석 유지
- 프로세스의 불변속성

---

# 오래된 주석 제거

오래된 주석은 없는 주석보다 위험합니다. 주석이 없으면 코드를 직접 읽게 되지만, 틀린 주석은 읽는 사람을 잘못된 방향으로 이끕니다.

```typescript
// 플레이어가 벽에 닿으면 게임을 종료한다
function handleCollision(player: Player, tile: Tile) {
  if (tile.isWall()) {
    player.takeDamage(1);
  }
}
```

주석은 "게임 종료"를 말하지만 코드는 "피해 적용"을 합니다. 둘 중 하나는 낡았습니다. 이때 문제는 단순히 문장이 틀렸다는 데 있지 않습니다. 읽는 사람은 주석과 코드 중 무엇을 믿어야 할지 판단해야 하고, 그 판단 비용이 코드 이해를 방해합니다.

오래된 주석은 보통 다음 상황에서 생깁니다.

- 코드 동작은 바뀌었지만 주석은 함께 수정되지 않았다.
- 임시 설명이 시간이 지나 사실처럼 남았다.
- 리팩터링으로 구조가 바뀌었는데 예전 설계 설명이 남았다.
- 주석이 구현 세부사항을 너무 자세히 설명해서 쉽게 낡았다.

따라서 주석을 볼 때는 먼저 이렇게 물어야 합니다.

> 이 주석은 지금 코드와 같은 사실을 말하고 있는가?

같은 사실을 말하지 않는다면 고쳐야 합니다. 그런데 대부분의 경우 더 나은 선택은 주석을 고치는 것이 아니라, **주석이 필요 없도록 코드를 바꾸는 것**입니다.

```typescript
function handleCollision(player: Player, tile: Tile) {
  if (tile.isWall()) {
    damagePlayer(player);
  }
}

function damagePlayer(player: Player) {
  player.takeDamage(1);
}
```

이제 "벽에 닿으면 피해를 준다"는 의도가 함수 이름과 호출 흐름에 드러납니다. 주석이 사실을 대신 설명하지 않아도 됩니다.

# 주석 처리된 코드 제거

주석 처리된 코드는 가장 먼저 삭제해도 되는 코드에 가깝습니다.

```typescript
function calculateScore(kills: number, bonus: number) {
  // const oldScore = kills * 10;
  // return oldScore + bonus;
  return kills * 12 + bonus;
}
```

이런 코드는 남겨둔 사람에게는 "혹시 필요할지도 모르는 백업"처럼 보입니다. 하지만 읽는 사람에게는 전혀 다르게 보입니다.

- 다시 살릴 예정인가?
- 지금 로직과 비교해야 하는가?
- 삭제하면 안 되는 중요한 히스토리인가?
- 왜 이 코드가 주석으로 남아 있는가?

즉, 주석 처리된 코드는 실행되지 않지만 여전히 읽는 사람의 주의를 빼앗습니다.

이 책의 관점에서 더 중요한 점은 버전 관리 시스템이 이미 과거 코드를 보관한다는 사실입니다. 과거 구현이 필요하다면 Git 히스토리에서 찾으면 됩니다. 현재 파일 안에는 **현재 동작하는 코드와 현재 필요한 설명**만 남아 있어야 합니다.

주석 처리된 코드는 다음 원칙으로 다룹니다.

- 실행되지 않는 코드는 삭제한다.
- 과거 구현은 버전 관리에 맡긴다.
- 실험 중인 코드는 별도 브랜치나 스파이크로 관리한다.
- 정말 다시 살릴 계획이 있다면 주석이 아니라 이슈나 작업 항목으로 남긴다.

코드 파일은 기억 창고가 아니라 실행 가능한 설계 문서입니다. 실행되지 않는 코드가 많아질수록 그 문서는 흐려집니다.

# 불필요한 주석 제거

좋은 주석은 코드가 말하지 못하는 것을 설명합니다. 반대로 불필요한 주석은 코드가 이미 말하고 있는 내용을 반복합니다.

```typescript
// 점수를 1 증가시킨다
score += 1;
```

이 주석은 아무 정보도 추가하지 않습니다. 오히려 읽는 사람은 코드와 주석을 모두 읽어야 하므로 비용만 늘어납니다.

불필요한 주석은 보통 다음 형태로 나타납니다.

```typescript
// 사용자를 가져온다
const user = getUser(id);

// 비어 있으면 false를 반환한다
if (items.length === 0) {
  return false;
}

// x 좌표를 설정한다
position.x = x;
```

이런 주석은 코드의 낮은 수준 동작을 다시 말합니다. 책이 계속 강조한 기준으로 보면, 문제는 주석이 아니라 **추상화 수준**입니다. 코드가 너무 낮은 수준으로 노출되어 있기 때문에 주석으로 의미를 붙이고 싶은 충동이 생깁니다.

불필요한 주석을 만났을 때는 삭제하기 전에 한 번 더 확인해야 합니다.

> 이 주석이 정말 중복인가, 아니면 코드에 드러나지 않은 의도를 말하고 있는가?

예를 들어 아래 주석은 단순 반복이 아닙니다.

```typescript
// 외부 정산 시스템이 음수 금액을 거부하므로 환불은 별도 API로 보낸다
if (amount < 0) {
  sendRefund(-amount);
  return;
}
```

코드만 보면 왜 음수를 따로 처리하는지 알기 어렵습니다. 이 주석은 구현이 아니라 **외부 시스템과의 제약**을 설명합니다. 이런 주석은 유지할 가치가 있습니다.

즉, 삭제 대상은 "주석"이 아니라 "중복 설명"입니다.

# 메서드의 이름으로 주석 대신하기

주석을 줄이는 가장 좋은 방법은 이름을 개선하는 것입니다. 특히 코드 블록 위에 주석이 붙어 있다면, 그 주석은 대개 메서드 이름으로 바꿀 수 있습니다.

```typescript
function update(player: Player) {
  // 체력이 0 이하이면 플레이어를 죽은 상태로 만든다
  if (player.health <= 0) {
    player.dead = true;
    player.animation = "dead";
  }
}
```

이 주석은 아래처럼 함수 이름으로 옮길 수 있습니다.

```typescript
function update(player: Player) {
  markDeadIfHealthIsDepleted(player);
}

function markDeadIfHealthIsDepleted(player: Player) {
  if (player.health <= 0) {
    player.dead = true;
    player.animation = "dead";
  }
}
```

이렇게 바꾸면 두 가지가 좋아집니다.

- `update()`를 읽는 사람은 세부 구현보다 의도를 먼저 본다.
- 세부 조건은 `markDeadIfHealthIsDepleted()` 안에 지역화된다.

책 전체의 흐름과도 연결됩니다. 긴 함수에서 주석은 종종 "여기부터 다른 생각이 시작된다"는 표시입니다. 그렇다면 그 표시를 그대로 함수 이름으로 승격하면 됩니다.

```typescript
function checkout(cart: Cart) {
  validateCart(cart);
  applyDiscounts(cart);
  chargePayment(cart);
  sendReceipt(cart);
}
```

이 코드에는 주석이 필요 없습니다. 각 줄이 이미 의도를 말하고 있기 때문입니다.

주석을 이름으로 바꿀 때 주의할 점도 있습니다. 이름이 너무 구현 중심이면 효과가 떨어집니다.

```typescript
function loopItemsAndSubtractAmount() {
  // ...
}
```

이 이름은 코드가 어떻게 동작하는지만 말합니다. 더 좋은 이름은 왜 그 일을 하는지 드러냅니다.

```typescript
function applyInventoryReduction() {
  // ...
}
```

좋은 이름은 주석을 줄이고, 좋은 주석은 이름으로 표현하기 어려운 맥락만 남깁니다.

### 계획을 위한 주석 사용

모든 주석을 즉시 없앨 수는 없습니다. 특히 리팩터링 중에는 아직 이름으로 옮기지 못한 생각을 잠시 표시해야 할 때가 있습니다.

```typescript
function migrateUsers(users: User[]) {
  // 1. 유효하지 않은 계정 제외
  // 2. 기존 권한을 새 권한 모델로 매핑
  // 3. 마이그레이션 결과 저장
}
```

이런 주석은 최종 설명이라기보다 작업 계획입니다. 책의 관점에서는 이 계획 주석을 그대로 오래 남겨두기보다, 각 단계를 메서드로 추출하는 출발점으로 삼는 것이 좋습니다.

```typescript
function migrateUsers(users: User[]) {
  const validUsers = excludeInvalidAccounts(users);
  const migratedUsers = mapToNewPermissionModel(validUsers);
  saveMigrationResult(migratedUsers);
}
```

계획 주석은 다음 조건에서 유용합니다.

- 아직 코드를 안전하게 나누기 전이다.
- 긴 함수를 이해하기 위해 임시로 덩어리를 표시한다.
- 팀원에게 작업 중인 의도를 짧게 남겨야 한다.
- 이후 리팩터링의 체크리스트로 사용할 수 있다.

하지만 계획 주석은 오래 머물수록 낡습니다. 리팩터링이 끝났다면 함수 이름, 타입, 테스트, 문서 중 더 적절한 위치로 옮기고 주석은 제거해야 합니다.

# 불변속성을 문서화한 주석 유지

책이 주석을 경계한다고 해서 모든 주석을 나쁘게 보지는 않습니다. 특히 **불변속성(invariant)** 을 설명하는 주석은 남길 가치가 있습니다.

불변속성은 코드가 항상 만족해야 하는 조건입니다.

```typescript
class RingBuffer {
  // start는 항상 end보다 작거나 같으며, end는 capacity를 넘지 않는다.
  private start = 0;
  private end = 0;
  private capacity: number;
}
```

이 주석은 단순히 필드가 무엇인지 설명하는 것이 아닙니다. 여러 필드 사이에 유지되어야 하는 관계를 설명합니다. 이런 정보는 이름만으로 표현하기 어렵습니다.

물론 더 좋은 방법은 불변속성을 코드 구조로 강제하는 것입니다. 예를 들어 잘못된 상태를 만들 수 없도록 생성자, 타입, 캡슐화를 사용할 수 있습니다.

하지만 현실의 코드에서는 모든 불변속성을 타입 시스템으로 표현하기 어렵습니다. 특히 다음과 같은 경우에는 주석이 실용적입니다.

- 여러 필드 사이의 관계를 설명해야 한다.
- 성능 최적화 때문에 직관적이지 않은 구조를 유지해야 한다.
- 외부 시스템의 제약 때문에 특정 순서를 지켜야 한다.
- 수학적 조건이나 도메인 규칙을 코드만으로 표현하기 어렵다.

이때 주석은 구현 설명이 아니라 **설계 제약의 문서화**가 됩니다.

좋은 불변속성 주석은 구체적이어야 합니다.

```typescript
// 나쁨: 캐시 관련 처리
private cache: Map<string, Result>;

// 좋음: cache에는 성공한 요청만 저장한다. 실패 응답은 재시도 정책을 위해 저장하지 않는다.
private cache: Map<string, Result>;
```

두 번째 주석은 코드를 바꾸는 사람이 지켜야 할 규칙을 알려줍니다. 이런 주석은 삭제하면 안 됩니다.

### 프로세스의 불변속성

불변속성은 데이터에만 존재하지 않습니다. 개발 프로세스에도 존재합니다.

예를 들어 다음과 같은 규칙은 코드 한 줄만 봐서는 알기 어렵습니다.

- 이 파일은 외부 도구가 생성하므로 직접 수정하면 안 된다.
- 이 마이그레이션은 배포 전에 한 번만 실행된다.
- 이 값은 레거시 클라이언트와의 호환성 때문에 유지된다.
- 이 순서는 운영 절차와 맞물려 있어 임의로 바꾸면 안 된다.

이런 내용은 코드 이름만으로 표현하기 어렵고, 테스트만으로도 충분히 전달되지 않을 수 있습니다. 이때 주석은 팀의 프로세스 지식을 코드 근처에 붙잡아두는 역할을 합니다.

```typescript
// 이 매핑은 v1 모바일 앱이 아직 사용하는 값이다. v1 지원 종료 전까지 이름을 바꾸지 않는다.
const legacyStatusNames = {
  waiting: "WAIT",
  complete: "DONE"
};
```

이 주석은 코드가 무엇을 하는지보다 **왜 지금 이 형태를 유지해야 하는지**를 설명합니다. 이런 "왜"는 주석이 가장 잘하는 일입니다.

8장을 실무적으로 정리하면 다음 기준이 남습니다.

- 코드가 이미 말하는 주석은 삭제한다.
- 코드가 말해야 하는 주석은 이름, 함수, 타입으로 옮긴다.
- 코드만으로 표현하기 어려운 이유와 제약은 주석으로 남긴다.
- 주석 처리된 코드는 버전 관리에 맡기고 삭제한다.
- 오래된 주석은 적극적으로 제거한다.

결국 주석을 줄인다는 것은 설명을 없애는 일이 아닙니다. 설명의 위치를 더 적절한 곳으로 옮기는 일입니다. 가능한 설명은 코드로 만들고, 코드가 담기 어려운 맥락만 주석으로 남기는 것이 이 장의 핵심입니다.
