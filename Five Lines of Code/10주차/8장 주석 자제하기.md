# 8장 주석 자제하기

좋은 코드는 주석이 많아서 이해되는 코드가 아니라, **이름과 구조만으로 의도가 드러나는 코드**입니다.

8장의 핵심은 "주석을 모두 없애라"가 아닙니다. 주석이 코드의 부족함을 덮고 있는지, 아니면 코드만으로 표현하기 어려운 중요한 지식을 보존하고 있는지를 구분하는 것입니다.

이번 장에서 다룰 내용
- 주석의 위험성 이해하기
- 가치를 더하는 주석 구별하기
- 다양한 유형의 주석 처리하기

---

# 오래된 주석 제거

오래된 주석은 없는 주석보다 위험합니다.

코드는 수정되었는데 주석은 그대로 남아 있으면, 읽는 사람은 코드와 주석 중 무엇을 믿어야 할지 판단해야 합니다. 이때 주석은 도움말이 아니라 **거짓 단서**가 됩니다.

오래된 주석을 발견했을 때의 기준은 단순합니다.

- 코드와 다르면 삭제합니다.
- 여전히 필요한 지식이라면 코드나 테스트로 옮깁니다.
- 판단이 어렵다면 주석을 믿지 말고 실제 동작을 확인합니다.

주석은 컴파일되지 않고, 테스트되지 않으며, 자동으로 최신 상태가 되지 않습니다. 그래서 코드보다 더 빨리 부패합니다.

---

# 주석 처리된 코드 제거

주석 처리된 코드는 대부분 삭제해야 합니다.

예전에는 "혹시 나중에 필요할지 모르니 남겨두자"는 이유로 코드를 주석 처리해 보관했습니다. 하지만 지금은 버전 관리 시스템이 그 역할을 합니다. 필요하면 Git 이력에서 다시 찾으면 됩니다.

주석 처리된 코드가 남아 있으면 다음 비용이 생깁니다.

- 현재 사용 중인 코드인지 판단해야 합니다.
- 검색 결과에 불필요하게 걸립니다.
- 리팩터링할 때 같이 고쳐야 하는지 헷갈립니다.
- 실제 코드의 의도를 흐립니다.

쓰지 않는 코드는 보관 대상이 아니라 삭제 대상입니다.

---

# 불필요한 주석 제거

코드를 그대로 반복하는 주석은 제거합니다.

```typescript
// 사용자를 저장한다
saveUser(user);
```

이런 주석은 새로운 정보를 주지 않습니다. 오히려 코드를 읽는 속도를 늦춥니다.

불필요한 주석의 특징은 다음과 같습니다.

- 코드만 읽어도 알 수 있는 내용을 설명합니다.
- 메서드명이나 변수명과 같은 말을 반복합니다.
- "무엇을 하는지"만 말하고 "왜 필요한지"는 말하지 않습니다.

주석이 필요해 보인다면 먼저 이름을 바꿀 수 있는지 확인해야 합니다. 이름으로 표현할 수 있는 정보는 주석보다 이름에 두는 편이 낫습니다.

---

# 메서드의 이름으로 주석 대신하기

주석이 코드 블록의 의도를 설명하고 있다면, 그 블록은 메서드로 추출할 후보입니다.

```typescript
// 사용자가 할인 대상인지 확인한다
if (user.age >= 65 || user.hasCoupon()) {
  price = price * 0.8;
}
```

이 경우 주석을 남기기보다 메서드 이름으로 의도를 드러낼 수 있습니다.

```typescript
if (isDiscountEligible(user)) {
  price = price * 0.8;
}
```

이렇게 바꾸면 주석은 사라지고, 코드 자체가 의도를 말합니다.

핵심은 다음과 같습니다.

- 주석이 코드 덩어리의 제목처럼 보이면 메서드로 추출합니다.
- 메서드명은 내부 구현이 아니라 호출자의 의도를 기준으로 짓습니다.
- 추출한 메서드가 너무 얇고 의미가 없다면 유지하지 않습니다.

좋은 이름은 가장 강력한 주석입니다.

### 계획을 위한 주석 사용

작업 중 임시로 남기는 계획 주석은 유용할 수 있습니다.

```typescript
// TODO: 여기서 인증 실패 케이스를 분리한다
```

다만 이런 주석은 개발 중 사고를 돕기 위한 메모입니다. 배포 전에는 코드, 테스트, 이슈 관리 도구 중 하나로 옮기거나 삭제해야 합니다.

계획 주석을 남겨도 되는 경우는 제한적입니다.

- 지금 당장 이어서 작업할 내용을 표시할 때
- 리팩터링 중 임시 단계가 필요할 때
- 팀이 명시적으로 추적하는 TODO 규칙이 있을 때

그 외의 TODO는 시간이 지나면 기술 부채가 됩니다.

# 불변속성을 문서화한 주석 유지

모든 주석이 나쁜 것은 아닙니다.

코드만으로 표현하기 어렵지만 반드시 지켜야 하는 규칙, 특히 **로컬이 아닌 불변속성**을 설명하는 주석은 남길 수 있습니다.

예를 들어 여러 클래스, 외부 시스템, 데이터베이스 제약이 함께 맞물려야 하는 규칙은 한 메서드 이름만으로 표현하기 어렵습니다.

```typescript
// 이 값은 외부 결제 시스템의 중복 요청 방지 키와 같아야 한다.
```

이런 주석은 단순 설명이 아니라 변경자가 알아야 할 제약을 알려줍니다.

다만 가능하다면 우선순위는 다음과 같습니다.

1. 타입이나 객체 구조로 표현합니다.
2. 자동 테스트로 고정합니다.
3. 그래도 어렵다면 주석으로 문서화합니다.

주석은 마지막 선택지입니다. 하지만 중요한 제약을 코드로 표현할 수 없다면, 침묵보다 정확한 주석이 낫습니다.

---

### 프로세스의 불변속성

프로세스의 불변속성은 코드 실행 순서나 운영 절차에 숨어 있는 규칙입니다.

예를 들면 다음과 같습니다.

- 배포 전에 특정 마이그레이션이 먼저 실행되어야 합니다.
- 캐시를 지우기 전에 새 설정이 반영되어야 합니다.
- 특정 API는 외부 시스템과 순서가 맞아야 합니다.

이런 지식은 코드 한 줄에 담기 어렵습니다. 그래서 문서화가 필요할 수 있습니다.

하지만 책의 관점에서는 먼저 질문해야 합니다.

> 이 절차를 사람이 기억하지 않아도 되게 만들 수 있는가?

가능하다면 스크립트, 테스트, 배포 파이프라인, 타입 구조로 옮기는 것이 더 낫습니다. 주석은 사람이 규칙을 지키도록 부탁하지만, 자동화는 규칙을 어기기 어렵게 만듭니다.

---

# 요약

- 주석은 개발 중에는 유용할 수 있지만 배포 전에는 제거해야 합니다.
- 주석에는 다섯 가지 유형이 있습니다.
	- 오래된 주석은 버그를 유발할 수 있으므로 삭제해야 합니다.
	- 버전 관리 시스템이 있으므로 주석 처리된 코드는 삭제해야 합니다.
	- 불필요한 주석은 가독성을 더하지 않기 때문에 삭제해야 합니다.
	- 메서드 이름이 될 수 있는 주석은 메서드명으로 대신합니다.
	- 로컬이 아닌 불변속성을 문서화하는 주석은 코드 또는 자동 테스트로 변환해야 합니다. 여의치 않으면 주석을 유지해도 좋습니다.
	- 프로세스의 불변속성은 가능하면 자동화하고, 어렵다면 정확하게 문서화합니다.

결국 주석을 줄이는 목적은 설명을 없애는 것이 아닙니다. **설명이 코드 구조 안에 들어가게 만드는 것**입니다.
