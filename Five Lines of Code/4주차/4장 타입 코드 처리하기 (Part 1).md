이번 파트에서 다룰 내용
- **if 문에서 else 를 사용하지 말 것**으로 이른 바인딩 이해하기
- **클래스로 타입 코드 대체**로 타입별 책임을 나눌 준비하기
- **클래스로의 코드 이관**으로 분기 로직을 객체 내부로 이동하기
- **메서드의 인라인화**로 가독성에 기여하지 않는 얇은 위임 제거하기



# 간단한 if 문 리팩터링

### 규칙: if 문에서 else를 사용하지 말 것
#### 정의
프로그램에서 이해하지 못하는 타입(형) 인지를 검사하지 않는 한 if 문에서 else를 사용하지 마십시오.

#### 설명
if-else를 사용하면 코드에서 결정이 내려지는 지점을 고정하게 됩니다. 그럴 경우 if-else가 있는 위치 이후에서는 다른 변형을 도입할 수 없기 때문에 코드의 유연성이 떨어집니다.

if-else 는 하드코딩된 결정으로 볼 수 있습니다. 코드에서 하드코딩된 상수가 좋지 않는 것처럼 하드코딩된 결정도 좋지 않습니다.

어떤 결정을 하드코딩하지 않는 것이 좋은데, 바로 if 구문을 else 구문과 함께 사용하지 않는 것입니다. 안타깝게도 이를 위해서는 우리가 무엇을 검사하는지에 주의를 기울여야 합니다. 예를들어 e.key 를 사용해서 어떤 키가 눌렸는지 검사하는데, 그것이 string 타입(자료형) 이라고 해봅시다. string 자료형의 구현을 바꿀 수 있다면 모를까, else if 체인 구문을 피할 수 없습니다. 

그러나 이런 경우는 일반적으로 사용자의 입력을 받거나 데이터베이스에서 값을 가져오는 등 애플리케이션 외부에서 입력을 받는 프로그램의 경계에서 발생하기 때문에 문제가 되지 않습니다. 이 경우 가장 먼저 할 일은 외부의 데이터 타입을 내부에서 제어 가능한 데이터 타입으로 매핑하는 것입니다. 

```
-- 예제 4.2 사용자 입력을 내부에서 제어 가능한 데이터 타입으로 매핑
window.addEventListener("keydown", e => {
	if (e.key === LEFT_KEY || e.key === "a") input.push(Input.LEFT);
	else if (e.key === UP_KEY || e.key === "w") input.push(Input.UP);
	else if (e.key === RIGHT_KEY || e.key === "d") input.push(Input.RIGHT);
	else if (e.key === DOWN_KEY || e.key === "s") input.push(Input.DOWN);
});
```

조건절에서 두 가지 데이터 타입(KeyboardEvent, string) 중 어느 것도 우리가 결정할 수 없습니다. 앞에서 언급한 것처럼 이러한 else if 체인 구문은 데이터의 입출력에 직접 연결되어야 하며 애플리케이션의 나머지 부분과는 분리되어야 합니다.

참고로 독립된 if 문은 검사로 간주하고, if-else 문은 의사결정으로 간주합니다. 이를 기준으로 다음 예시와 같이 조기에 return 값을 결정하기 어려운 메서드에서는 시작 시에 간단한 검증을 수행할 수 있습니다. 따라서 이 규칙은 특별히 else를 대상으로 합니다.

그 외에도 이 규칙은 쉽게 검증할 수 있습니다. 그냥 else 문을 찾으면 됩니다.
```
--예제 4.3 변경 전
function average(ar: number[]) {
	if (size(ar) === 0) 
		throw "Empty array not allowed";
	else
		return sum(ar) / size(ar);
}
```

```
--예제 4.4 변경 후
function assertNotEmpty(ar: number[]) {
	if (size(ar) === 0)
		throw "Empty array not allowed";
}
function average(ar: number[]) {
	assertNotEmpty(ar);
	return sum(ar) / size(ar);
}
```

#### 스멜
이 규칙은 스멜로 인식되는 **이른 바인딩(early binding)** 과 관련 있습니다. 프로그램을 컴파일할 때 if-else 같은 의사결정 동작은 컴파일 시 처리되어 애플리케이션에 고정되며 재컴파일 없이는 수정할 수 없습니다. 이것의 반대는 코드가 실행되는 순간에 동작이 결정되는 **늦은 바인딩(late binding)** 입니다.
이른 바인딩은 if 문을 수정해야 변경할 수 있기 때문에 추가에 의한 변경을 방해합니다. 늦은 바인딩의 속성은 2장에서 배운 것 같은 추가를 통한 변경을 가능하게 합니다.

#### 의도
if는 조건 연산자로 흐름을 제어합니다. 이것은 다음에 실행할 코드를 결정한다는 것을 의미합니다. 그러나 객체지향 프로그래밍에는 객체라는 훨씬 더 강력한 제어 흐름 연산자가 있습니다. 인터페이스를 사용한 두 가지 다른 구현이 있는 경우 인스턴스화하는 클래스에 따라 실행할 코드를 결정할 수 있습니다. 본질적으로 이 규칙은 더 유연한 도구인 객체를 사용하는 방안을 찾게 합니다.



### 리팩터링 패턴: 클래스로 타입 코드 대체

#### 설명
이 리팩터링 패턴은 열거형을 인터페이스로 변환하고 열거형의 값들은 클래스가 됩니다. 그렇게 하면 각 값에 속성을 추가하고 해당 특정 값과 관련된 기능을 특성에 맞게 만들 수 있습니다. 이것은 다음에 논의할 또 다른 리팩터링 패턴인 클래스로 코드 이관과 함께 사용해서 추가를 통한 변경으로 이어집니다. 그 이유는 애플리케이션 전체에 걸쳐 switch 또는 else if 체인으로 열거형을 자주 사용하기 때문입니다. switch 는 열거형이 가지는 값들이 처리되어야 하는 방식을 한곳에 기술한 것입니다.

열거형의 값을 클래스로 변환할 때는 다른 열거 값을 고려하지 않고 해당 값과 관련된 기능을 그룹화할 수 있습니다. 이 프로세스는 기능과 데이터를 함께 제공하는데, 기능을 해당 값의 특징에 맞게 만들 수 있습니다. 열거형에 새 값을 추가하는 것은 수많은 파일에 걸쳐서 해당 열거형과 연결된 로직들을 확인해야 하는 반면, 인터페이스를 구현한 새로운 클래스를 추가하는 것은 해당 클래스에 메서드의 구현이 필요할 뿐, 새로운 클래스를 사용하기 전까지는 다른 코드를 수정하지 않아도 됩니다.

타입 코드 또한 열거형이 아닌 형태로 변환할 수 있습니다. 정수 타입 또는 일치 비교 연산자 === 를 지원하는 모든 타입은 타입 코드로 볼 수 있습니다. 가장 일반적으로 int와 enum을 사용합니다. 다음은 티셔츠 크기에 대한 타입 코드의 예입니다.

```
--예제 4.15 초기 코드
const SMALL = 33;
const MEDIUM = 37;
const LARGE = 42;
```

정해진 상수를 참조하지 않고 숫자를 사용했을 수 있기 때문에 int의 경우 타입 코드의 사용을 추적하는 것이 더 까다롭습니다. 따라서 타입 코드를 볼 때는 항상 즉시 열거형으로 변환합니다. 그래야만 이 리팩터링 패턴을 안전하게 적용할 수 있습니다.

```
-- 예제 4.16 변경 전
const SMALL = 33;
const MEDIUM = 37;
const LARGE = 42;
```

```
--예제 4.17 변경 후
enum TShirtSizes {
	SMALL = 33,
	MEDIUM = 37,
	LARGE = 42
}
```

이 리팩터링 패턴은 자체적으로 많은 가치를 가지지는 않지만 나중에 환상적인 개선을 가능하게 합니다. 모든 값에 대한 메서드를 갖는 것도 스멜이기 때문에 이것은 하나의 스멜을 다른 스멜로 대체한 것입니다. 하지만 열거형 값들은 밀접하게 연결되어 있는 반면, 메서드는 하나씩 처리할 수 있습니다.

#### 예제
타입 코드가 값의 집합으로만 존재하면, 기능은 바깥의 `if` 나 `switch` 로 흘러가기 쉽습니다.

```
-- 예제 4.18 변경 전
enum TShirtSize {
	SMALL = 33,
	MEDIUM = 37,
	LARGE = 42
}
```

이 상태에서는 값은 있지만 책임은 없습니다. 이제 타입 코드를 클래스로 바꾸면 각 값이 자신의 의미를 직접 가질 수 있습니다.

```
-- 예제 4.19 변경 후
interface TShirtSize {
	chest(): number;
	label(): string;
}

class Small implements TShirtSize {
	chest() { return 33; }
	label() { return "S"; }
}

class Medium implements TShirtSize {
	chest() { return 37; }
	label() { return "M"; }
}

class Large implements TShirtSize {
	chest() { return 42; }
	label() { return "L"; }
}
```

이 시점의 핵심은 "클래스가 더 길어졌다"가 아닙니다. 이제부터는 `SMALL` 일 때의 정책, `MEDIUM` 일 때의 정책을 외부 분기문이 아니라 각 클래스 안으로 옮길 수 있게 되었다는 점이 중요합니다.


### 리팩터링 패턴: 클래스로의 코드 이관

#### 설명
이 리팩터링 패턴은 기능을 클래스로 옮기기 때문에 클래스로 타입 코드 대체 패턴의 자연스러운 연장선에 있습니다. 결과적으로 if 구문이 제거되고 기능이 데이터에 더 가까이 이동합니다. 특정 값과 연결된 기능이 값에 해당하는 클래스로 이동하기 때문에 이는 불변속성을 지역화하는데 도움이 됩니다.

가장 단순한 형태로, 항상 메서드 전체를 클래스로 옮긴다고 가정합니다. 코드를 먼저 추출하지 않고 바로 옮기는 것도 가능하지만, 그렇게되면 그 사이에 문제가 없는지 확인하는데 더 많은 주의가 필요합니다.

#### 절차
1. 원래 함수를 복사하여 모든 클래스를 붙여 넣습니다. 이제 메서드이므로 function 키워드를 제거합니다. 컨텍스트를 this로 바꾼 후 사용하지 않는 매개변수를 제거합니다. 메서드에 여전히 잘못된 이름이 존재하기 때문에 오류가 계속 발생합니다.
2. 메서드 선언 부분의 메서드 이름과 매개변수 리스트(메서드 시그니처)를 인터페이스에 복사하고 원래 메서드와 약간 다른 이름을 지정합니다.
3. 모든 클래스에서 새로운 메서드를 점검합니다.
   1. 클래스에 맞게 조건식들의 true, false 를 결정합니다.
   2. 미리 계산할 수 있는 모든 계산을 수행합니다. 일반적으로 if (true) 와 if (false) {...} 를 제거하는 것과 같지만 먼저 조건을 단순화해야 할 수도 있습니다.(예: false || true 는 true)
   3. 메서드 처리가 완료되었음을 알릴 수 있는 적절한 이름으로 변경합니다. 이때 컴파일러에서 에러가 발생하면 안됩니다.
4. 원래 함수의 본문을 새로운 메서드에 대한 호출로 바꿉니다.

#### 예제
다음 코드는 타입 코드에 따라 배송비를 분기합니다. 지금은 모든 정책이 `switch` 안에 모여 있습니다.

```
-- 예제 4.20 변경 전
enum TShirtSize {
	SMALL,
	MEDIUM,
	LARGE
}

function shippingCost(size: TShirtSize): number {
	switch (size) {
		case TShirtSize.SMALL:
			return 2500;
		case TShirtSize.MEDIUM:
			return 3000;
		case TShirtSize.LARGE:
			return 3500;
	}
}
```

이 로직을 각 타입으로 옮기면 분기 대신 다형성을 사용하게 됩니다.

```
-- 예제 4.21 변경 후
interface TShirtSize {
	shippingCost(): number;
}

class Small implements TShirtSize {
	shippingCost() { return 2500; }
}

class Medium implements TShirtSize {
	shippingCost() { return 3000; }
}

class Large implements TShirtSize {
	shippingCost() { return 3500; }
}

function shippingCost(size: TShirtSize): number {
	return size.shippingCost();
}
```

이후 `XLarge` 가 추가되더라도 기존 `switch` 를 열어 수정할 필요가 없습니다. 새로운 클래스를 추가하고 필요한 시점에 그 객체를 주입하면 됩니다. 이것이 이 책이 강조하는 "수정보다 추가"의 대표적인 예입니다.




### 불필요한 메서드 인라인화

가독성에 도움이 되지 않는 함수 제거하기

```
-- 예제 4.52 변경 전
handleInput(current);
```

```
-- 예제 4.53 변경 후
current.handle();
```

이후 current를 input 으로 이름을 바꾸고 나면 handleInput 의 모양은 다음과 같습니다.

```
-- 예제 4.54 변경 전
function handleInputs() {
	while (inputs.length > 0) {
		let current = inputs.pop();
		handleInput(current);
	}
}

function handleInput(input: Input) {
	input.handle();
}
```

```
-- 예제 4.55 변경 후
function handleInputs() {
	while (inputs.length > 0) {
		let input = inputs.pop();
		input.handle();  // 메서드의 인라인화
	}
}

// 삭제된 handleInput
```

**메서드의 인라인화** 리팩터링 패턴은 3장의 **메서드 추출**과 정확히 반대입니다.


### 리팩터링 패턴: 메서드의 인라인화

#### 설명
두 가지 중요한 주제는 코드 추가(클래스를 지원하기 위해)와 코드 제거 입니다. 이 리팩터링 패턴은 후자를 지원합니다. 즉, 프로그램에서 더 이상 가독성에 도움이 되지 않는 메서드를 제거합니다. 그 방법은 메서드에서 이를 호출하는 모든 곳으로 코드를 옮기는 것입니다. 이렇게 하면 메서드가 사용되지 않으므로 안전하게 삭제할 수 있습니다.

**메서드의 인라인화** 리팩터링 패턴과 메서드를 인라인으로 사용하는 것은 다르다는 점에 유의하십시오. 이전 절에서 코드를 클래스로 이관할 때 is 로 시작하는 메서드를 인라인으로 사용했습니다. 그러고 나서 원래의 함수를 제거하기 위해 메서드를 인라인화했습니다. 메서드를 인라인으로 사용할 때는 모든 호출 측을 수정하지 않으므로 원래의 메서드를 유지합니다. 이것은 일반적으로 호출하는 측을 단순화하기 위한 것입니다. **메서드의 인라인화**를 수행할 때는 모든 호출 측을 수정하여 원래의 메서드를 제거합니다.

흔히 메서드가 한 줄만 있는 경우 이 작업을 수행합니다. 이것은 엄격한 **다섯 줄 제한** 규칙 때문인데, 한 줄인 메서드를 인라인화하면 이 규칙을 위반하지 않습니다. 이 리팩터링 패턴을 한 줄 이상인 메서드에 적용할 수도 있습니다.

또 다른 고려사항은 메서드가 인라인화하기에 너무 복잡하지 않은지 여부입니다. 다음에 소개하는 메서드는 숫자의 절댓값을 제공합니다. 성능에 최적화되었기 때문에 분기가 없습니다. 한 줄입니다. 목적을 위해 낮은 수준의 연산에 의존하며 메서드로 존재하는 것이 가독성에 도움이 되기 때문에 이것을 인라인화해서는 안됩니다. 이 경우 인라인화하면 '작업은 동일한 추상화 수준에 있어야 한다'에 반하는 스멜이 생기는데, 이는 **호출 또는 전달, 한 가지만 할 것 규칙**을 적용해야하는 이유가 됩니다.

#### 예제

```
-- 예제 4.56 유지해야 하는 메서드
function abs(x: number) {
	let mask = x >> 31;
	return (x + mask) ^ mask;
}

function averageDelta(delta: number[]) {
	return abs(sum(delta)) / size(delta);
}
```

`abs` 는 한 줄짜리 계산처럼 보여도, 실제로는 비트 연산을 이용한 저수준 구현 세부사항을 숨기고 있습니다. 이 메서드를 인라인화하면 호출부는 이렇게 변합니다.

```
-- 예제 4.57 인라인화하면 오히려 나빠지는 경우
function averageDelta(delta: number[]) {
	let total = sum(delta);
	let mask = total >> 31;
	return ((total + mask) ^ mask) / size(delta);
}
```

이 버전은 함수 수는 줄었지만 가독성은 분명히 나빠졌습니다. 즉, **메서드의 인라인화는 얇은 위임을 제거할 때만 유효**하고, 저수준 구현을 숨기는 경계까지 없애면 안 됩니다.

# 요약
- `if-else` 는 결정을 코드 위치에 고정하기 때문에, 가능한 한 경계에서만 사용하고 내부에서는 늦은 바인딩으로 바꾸는 것이 좋습니다.
- `클래스로 타입 코드 대체`는 단순히 enum을 클래스로 바꾸는 작업이 아니라, 타입별 책임을 각 객체에 배치할 수 있게 만드는 준비 단계입니다.
- `클래스로의 코드 이관`은 `if` 와 `switch` 를 제거하고, "타입에 따라 달라지는 동작"을 타입 자신에게 맡기는 과정입니다.
- `메서드의 인라인화`는 불필요한 위임을 걷어낼 때 유용하지만, 추상화를 제공하는 메서드까지 제거해서는 안 됩니다.
